## 一、阶段愿景

将 SPMA 从"被动问答系统"升级为**主动式知识发现引擎**——不仅仅回答用户的问题，而是主动感知知识变化、跨源推演影响链、记忆用户偏好，成为一个真正的"企业知识助手"而非"企业知识搜索引擎"。

### 核心能力方向

| 能力 | 说明 | 用户价值 |
|------|------|---------|
| **主动感知** | 当 PRD 文档更新或代码变更时，自动通知可能受影响的用户 | "有人改了支付模块的 PRD，你可能需要知道" |
| **跨源推演** | 自动分析"这段代码改了会影响哪些表和文档" | 不等人问，提前预判影响面 |
| **用户记忆** | 记住用户的角色、关注模块、历史问题，个性化排序和推荐 | "你上次关注过这个需求的进展" |
| **Agent 间网状互调** | Doc Agent 发现线索后主动调 Code Agent，而非等 Supervisor 调度 | 更自然的跨源关联发现 |

## 三、可能的功能方向（待调研后细化）

### 3.1 主动感知（Proactive Awareness）

**场景示例：**
> PM 修改了"支付模块"的 PRD → 系统检测到变更 → 自动分析：这个 PRD 关联的 `req_id=REQ-2024-0187` 在代码中被 `payment-service` 仓库引用 → 通知 payment-service 的开发者："REQ-2024-0187 的需求文档有更新，可能影响你维护的 payment-service"

**技术方向：**
- 事件总线：摄入管道变更事件 → 规则引擎 → 匹配用户关注 → 推送通知
- 用户关注模型：基于查询历史自动学习关注模块
- 通知渠道：企业 IM（飞书/企业微信/Slack）Webhook

### 3.2 跨源推演（Cross-Source Reasoning）

**场景示例：**
> 用户问"如果我把 users 表的 email 字段改成可空，会影响什么？" → 系统自动执行：
> 1. SQL Agent：查 users.email 的当前约束和关联表
> 2. Code Agent：搜所有引用 `users.email` 的代码
> 3. Doc Agent：搜相关 PRD 文档中关于 email 字段的要求
> 4. 推演引擎：综合三源信息 → 输出"会影响 3 个服务、需要修改 5 个文件、与 REQ-187 的需求冲突"

**技术方向：**
- 因果图构建：从代码调用链 + 外键依赖 + 需求关联中提取因果关系
- 影响面分析引擎：改一个字段 → 传播到所有依赖
- 与 Phase 5+ 的"Agent 间网状互调"配合使用

### 3.3 用户记忆（User Memory）

**场景示例：**
> 开发工程师张三连续三天查询"支付回调"相关问题 → 系统记住：
> 1. 张三关注支付模块
> 2. 当他问"那个接口改了没"时，系统知道"那个"指支付回调接口
> 3. 当支付相关 PRD 更新时，主动推送给张三

**技术方向：**
- 用户画像构建：角色（PM/开发/DBA）、关注模块、查询偏好
- 个性化检索：相同 query 对不同用户返回不同排序
- 长期记忆：跨 Session 记住用户上下文（非 Session 级别的短期记忆）
- 隐私保护：记忆数据不出企业边界，用户可随时清除

### 3.4 Agent 间网状互调（Mesh Agent Interaction）

**当前（Phase 3-4）：** Agent 间通过 Supervisor 的 DAG 编排——Supervisor 统一调度，Worker 之间不直接通信。

**Phase 5+ 方向：**
```
Doc Agent 发现线索 "REQ-187 关联表 users"
    │
    │ 主动发起（而非等 Supervisor 调度）
    ▼
SQL Agent: 查 users 表结构
    │
    │ 发现 "users.oauth_provider 列有枚举值限制"
    ▼
Code Agent: 搜 oauth_provider 的代码引用
```

**技术方向：**
- Agent-to-Agent 消息协议：Worker 之间直接发送"线索请求"
- 线索质量评估：Worker 需要判断发现的线索是否值得发起跨 Agent 请求（避免噪声放大）
- 与 Supervisor 的关系：Supervisor 从"编排者"演变为"仲裁者"——当 Worker 间互调产生矛盾时介入

## 四、NOT in Scope（明确不做）

以下方向在当前架构下不适合或 ROI 过低：

| 方向 | 原因 |
|------|------|
| 知识图谱混合方案 | 维护成本极高（需持续 CDC 同步），ROI 不明确 |
| 代码 Embedding 语义搜索 | Code Agent 使用 ripgrep 实时搜索，不建代码嵌入索引（设计决策） |
| 多模态输入（图片/语音/视频） | 企业内用户需求以文本为主，多模态 ROI 低 |
| 外部互联网知识融合 | 数据安全约束——敏感数据不出企业边界 |
| 自动代码修复/PR 生成 | 超出"知识问答"范围，进入"自动化开发"领域 |

---

## 五、实施建议

1. **先跑一个"主动感知"原型（4-6 周）：** 在 Phase 4 稳定后，选一个最高频的场景（如"PRD 变更通知开发者"），用最小成本验证用户价值
2. **用户记忆先做"短期记忆增强"（Phase 4 就能做）：** 上下文感知改写已经解决了 Session 内的指代消解，下一步是把 Session 内的记忆跨 Session 持久化
3. **Agent 间网状互调是最高风险的方向：** 可能引入"级联噪声"——一个 Agent 的错误线索被放大到其他 Agent。需要先设计严格的置信度门禁
4. **在 Phase 0-4 的用户反馈中收集 Phase 5+ 的需求信号：** 用户最常抱怨什么？"如果能主动告诉我 X 就好了"——这才是 Phase 5+ 应该优先做的
