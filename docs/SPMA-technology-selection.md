## 二、Agent 框架选型

### 2.1 LangChain 1.x 与 LangGraph 的关系

LangChain Inc. 于 **2025 年 10 月 22 日** 同时发布 LangChain 1.0 和 LangGraph 1.0 GA。**两者不是竞争关系，而是同一技术栈的不同抽象层次：**

```
LangChain 1.x (高抽象层)
  ├─ create_agent() — 标准 Tool-Calling Agent，一行代码创建
  ├─ Middleware — 可插拔横切关注点 (摘要/HITL/脱敏)
  ├─ 标准内容块 — 跨 Provider 统一输出格式
  ├─ 600+ 集成 — 模型/向量库/文档加载器
  └─ 底层运行在 LangGraph 运行时之上
         │
         ▼
LangGraph (低抽象层 / 运行时)
  ├─ StateGraph — 显式状态机，自定义节点+边+条件分支
  ├─ Checkpointer — 每步自动快照，服务重启后精确恢复
  ├─ Send API — 并行派发 + fan-in 收集 (多 Agent 编排核心)
  ├─ Human-in-the-Loop — interrupt_before/after 原语
  └─ 时间旅行调试 — 回退到任意历史状态
```

**SPMA 的选型策略：LangChain 1.x 作为框架层 + LangGraph 作为运行时层，两者配合使用。**

- **80% 的代码**使用 LangChain 1.x 的高层抽象：`create_agent` 创建各 Worker Agent、Middleware 注入脱敏/摘要、标准内容块统一 Claude/OpenAI 输出
- **20% 的场景**（Supervisor 并行派发、复杂收敛契约、Checkpointer namespace 隔离）下沉到 LangGraph StateGraph 做精确控制

### 2.3 选型结论

**首选：LangChain 1.x（框架层）+ LangGraph（运行时层）**

SPMA 5 Agent 架构需要两个层次的支撑：

**框架层（LangChain 1.x）提供：**

1. **`create_agent`** —— 一行代码创建具备 ReAct 循环的标准 Agent。Doc/Code/SQL/Synthesis Agent 各自是一个 `create_agent` 实例，内部自动执行"推理→工具调用→观察→推理"循环
2. **Middleware 体系** —— 6 个标准 Hook 点（`before_agent → before_model → wrap_model_call → wrap_tool_call → after_model → after_agent`），可插拔注入脱敏、摘要压缩、人工审核等横切关注点。每个 Agent 可以组合不同的 Middleware 链
3. **标准内容块** —— 跨 Claude/OpenAI 统一输出 citations、reasoning traces、tool calls，Synthesis Agent 的引用完整性检查依赖此能力
4. **动态模型选择** —— 运行时根据 state 自动切换模型：简单任务走 Haiku、复杂推理走 Sonnet。与 SPMA 的 LLM 分层策略天然匹配

**运行时层（LangGraph）提供：**

1. **StateGraph** —— 当 `create_agent` 内置的 ReAct 循环不够用时（如 SQL Agent 的 generate→guard→execute→verify 四阶段流水线），下沉到 StateGraph 自定义节点和条件边
2. **Send API** —— Supervisor 向 Doc/Code/SQL Agent 并行派发 + fan-in 收集的核心能力
3. **Checkpointer** —— 每个 Agent 子图独立 namespace 持久化，与 SPMA 的 Checkpointer 隔离设计完全吻合
4. **收敛契约** —— 条件边 + max_rounds 天然映射到 SPMA 的确定性收敛优先策略

**开发路径：先用 `create_agent` 快速搭建每个 Worker Agent，当中某个 Agent 的循环逻辑超出 ReAct 范式的表达力时，再将该 Agent 下沉为 StateGraph 定制。**

### 3.3 LLM 推理部署

**首选：vLLM（本地模型 Qwen3-8B/32B）**

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 推理引擎 | vLLM v0.6+ (V1 Engine) | PagedAttention KV 缓存管理，支持 prefix caching |
| GPU | 1× A100 80GB (8B) / 2-4× A100 80GB (32B TP) | 企业数据中心级 GPU |
| 量化 | FP8 (Hopper GPU) 或 AWQ INT4 | 几乎无损吞吐量翻倍 |
| 并发 | 8B: 50+ 并发; 32B: 20+ 并发 | 需实测调优 |
| Docker | `vllm/vllm-openai` 固定版本（不用 latest） | 生产必须 pin 版本 |
| K8s | HPA 基于 `vllm:num_requests_running`（非 CPU） | 比 CPU HPA 反应更快 |

**关键参数：**
- `--enable-prefix-caching`：Agent 循环场景下共享 system prompt，显著降低 TTFT
- `--gpu-memory-utilization 0.90`：平衡并发容量与 OOM 风险
- `--max-model-len 8192`：低于模型最大上下文，释放 VRAM 给更大 batch size

| **运维复杂度** | ⭐ 极低（复用PG）| ⭐⭐ 中 | ⭐⭐ 中 | ⭐⭐⭐⭐ 高 |
| **最大规模** | 千万-亿级 | 十亿级 | 亿级 | 千亿-万亿级 |
| **中文支持** | ✅ PG 原生 | ✅ | ✅ | ✅ |
| **成本** | 零新增（已有 PG）| 需新部署 | 需新部署 | 需 K8s+etcd+MinIO+Kafka |
| **ACID 事务** | ✅ | ❌ | ❌ | ❌ |

### 4.2 选型结论

**首选：PGVector (HNSW 索引)，未来扩容路径：Qdrant**

选择 PGVector 的核心逻辑：

1. **复用现有 PostgreSQL**——企业环境已有 PG 集群和运维能力
2. **向量+元数据 SQL JOIN**——SPMA 场景中频繁需要 `req_id`、`time_range` 等标量过滤与向量搜索在同一事务中
3. **千万级向量足够**——以平均 500 tokens/chunk 计，千万级向量覆盖 ~50 亿字的文档，远超企业内 PRD 和 Schema 的数据量
4. **ACID 事务**——数据摄入时的原子性保证（更新文档→删除旧 chunks→插入新 chunks 在一个事务中）

**迁移策略：** 当向量数量突破 5000 万或 P99 延迟持续超过 500ms 时，拆分出 Qdrant 专门服务向量检索，PG 保留元数据和标量查询。

### 4.3 PGVector 配置建议

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 索引类型 | HNSW | 构建快、查询性能好 |
| `m` (HNSW) | 16 | 连接数，影响精度和构建速度 |
| `ef_construction` | 200 | 构建时搜索深度 |
| `ef_search` | 100 | 查询时搜索深度，可动态调整 |
| 向量维度 | 1024 | 匹配 BGE-M3 |
| 距离度量 | cosine | 文本语义相似度 |
| 连接池 | 20+ 连接 | 并行检索需要 |

### 5.1 行业对比

| 维度 | **BGE-M3** ✅ | OpenAI text-embedding-3 | Cohere Embed v3 | Voyage AI v3 |
|------|-------------|------------------------|-----------------|--------------|
| **开源** | ✅ MIT | ❌ API only | ❌ API only | ❌ API only |
| **中文能力** | ⭐⭐⭐⭐⭐ 顶级 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **检索方式** | Dense+Sparse+ColBERT 三合一 | Dense only | Dense only | Dense (MoE) |
| **向量维度** | 1024 | 512/1536 | 1024 | 1024/2048 |
| **本地部署** | ✅ | ❌ | ❌ | ❌ |
| **成本** | 零 API 成本（需 GPU）| $0.00002-0.00013/K tokens | ~$0.0001/K tokens | $0.06-0.12/M tokens |
| **数据出境风险** | 零 | 需出境 | 需出境 | 需出境 |
| **MTEB Retrieval** | 中文 SOTA | 英文 SOTA | 多语言均衡 | 头部但中文评测少 |

### 5.2 选型结论

**首选：BGE-M3（本地部署）**

选择 BGE-M3 的决定性因素：

1. **数据不出企业边界**——这是 SPMA 的核心约束，LLM 调用经过脱敏层，Embedding 同样不能在外部 API 上运行
2. **Dense + Sparse + ColBERT 三合一**——Dense 用于语义搜索，Sparse（类似 BM25）用于精确术语匹配，天然支持混合检索，这正是 Doc Agent 需要的
3. **中文是绝对优势**——BAAI（智源研究院）出品，中文语义理解社区公认最强
4. **可微调**——在企业的 PRD 文档和数据库 Schema 上做 domain adaptation 后，召回率可超越商业 API（某电商案例：75%→95%）

**基础设施需求：**
- GPU：1× A100 80GB（支持批量嵌入，~50 docs/sec），或 1× RTX 4090 24GB（开发测试，~20 docs/sec）
- 推理框架：vLLM 或 Sentence-Transformers
- 部署：K8s 独立 Service，水平扩展

**备选思路（混合）：** 80% 流量走本地 BGE-M3，20% 冷门长尾场景可选 Cohere Embed v3 做 A/B 验证。

## 六、检索引擎选型

### 6.1 Doc Agent：混合检索引擎

Doc Agent 需要 BM25 + 向量混合检索，这需要两个引擎协作：

**方案组合：**

```
Doc Agent 检索架构
═════════════════════
                    ┌──────────────────────┐
  检索请求          │    Doc Agent         │
                    │    (LangChain Agent)   │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
              ▼                ▼                ▼
    ┌─────────────────┐ ┌───────────┐ ┌──────────────────┐
    │  Elasticsearch   │ │ PGVector  │ │   PostgreSQL     │
    │  BM25 关键词检索 │ │ 向量检索   │ │   元数据过滤       │
    │  Top-20          │ │ Top-20    │ │ WHERE req_id/     │
    │  + 中文分词      │ │ HNSW      │ │ time_range/       │
    │                  │ │           │ │ doc_types         │
    └────────┬────────┘ └─────┬─────┘ └────────┬─────────┘
             │                │                │
             └────────────────┼────────────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │ RRF 加权融合      │
                    │ Top-10 返回       │
                    └──────────────────┘
```

### 6.2 BM25 引擎选型

| 维度 | **Elasticsearch** ✅ | 自研 BM25 (SQLite FTS) | Meilisearch | Typesense |
|------|-------------------|----------------------|-------------|-----------|
| **中文分词** | IK/ANSJ/Jieba (成熟) | ❌ 需自建 | 内置 zh tokenizer | Jieba (旧版) |
| **BM25 NDCG@10** | 0.855 | ~0.82 (调优后) | **0.875** | 0.865 |
| **P99 延迟** | ~89ms | ~5ms | ~24ms | **~15ms** |
| **运维复杂度** | 中（需 JVM 调优）| 极低 | 极低 | 极低 |
| **规模** | 十亿级 | 百万级 | 千万级 | 千万级 |
| **向量混合** | ✅ kNN | ❌ | ❌ | ✅ HNSW |
| **成本** | 中 | 零 | 低 | 低 |
| **多语言** | 40+ 插件 | 依赖 SQLite | 30+ 内置 | 100+ 内置 |

**选型结论：**

- **Phase 1-2（< 100 万文档片段）：** 使用 PG 内置 `tsvector` + `tsquery` 做轻量 BM25。零新服务、零运维成本，SQL JOIN 元数据过滤天然一体。中文分词可通过 `zhparser` 扩展支持
- **Phase 3+（规模增长）：** 引入 Elasticsearch 专门做 BM25。ES 的中文分词生态成熟（IK Analyzer），且支持 kNN 向量搜索，可与 PGVector 并行使用作为多层检索策略
- **不推荐 Meilisearch/Typesense**：虽然延迟极低、上手简单，但中文分词不如 ES 成熟，且缺少向量+关键词混合检索的原生能力

**为什么需要独立的 BM25 引擎？**

Doc Agent 的设计使用 RRF 将 BM25 和向量检索结果融合。PGVector 的 `tsvector` 可以做 BM25-like 检索，但：
- PG 的全文搜索使用 TF-IDF 变体而非纯 BM25，精度略低
- 无法利用倒排索引的高效词组/短语搜索
- 大规模（千万级文档）下性能低于专用检索引擎

Phase 1 用 PG 内置全文搜索可以满足 MVP 需求，但设计上预留 ES 切换接口。

### 6.3 Code Agent：零索引实时搜索

Code Agent **不使用嵌入索引和全文检索引擎**，而是：

| 工具 | 用途 | 延迟 |
|------|------|------|
| **ripgrep (rg)** | 代码内容实时搜索 | ~10ms |
| **glob** | 文件路径模式发现 | ~5ms |
| **read_file** | 完整文件上下文 | ~5ms |
| **git log** | 需求追溯/人员时间查询 | ~50ms |
| **TreeSitter** | AST 调用图解析 | ~20ms |

这些是系统级命令行工具，不需要中间件。详细设计见 [SPMA-design-03-code-worker.md](../SPMA-design-03-code-worker.md)。

## 七、内存数据库 / 缓存选型

### 7.1 行业对比

| 维度 | **Redis** ✅ | DragonflyDB | Valkey | KeyDB |
|------|------------|-------------|--------|-------|
| **吞吐量 (SET/GET)** | 180K-420K ops/s | **312K-1.2M ops/s** | 200K-450K ops/s | 245K-500K ops/s |
| **P95 延迟 (读)** | **~245μs** | ~298μs | ~260μs | ~445μs |
| **P99.9 尾延迟** | 3,456μs | **1,234μs** | 2,800μs | 4,123μs |
| **内存效率 (20M keys)** | ~4.2 GB | **~880 MB** | ~3.8 GB | ~4.5 GB |
| **多核扩展** | 有限 (单线程执行) | 线性 (shared-nothing) | 有限 | 好 (多线程) |
| **生态与模块** | ⭐⭐⭐⭐⭐ 最成熟 (RedisJSON/Search/Graph/Bloom/Streams) | ⭐⭐ ~95% 兼容 | ⭐⭐⭐⭐ 100% 兼容 | ⭐⭐⭐⭐ 100% 兼容 |
| **运维工具** | ⭐⭐⭐⭐⭐ Sentinel/Cluster/可视化/Dashboards | ⭐⭐ 成长中 | ⭐⭐⭐ 继承 Redis 生态 | ⭐⭐⭐ |
| **License** | RSAL/SSPL (2024 变更) | BSL 1.1 | BSD-3 | BSD-3 |
| **持久化** | RDB + AOF | RDB + WAL | RDB + AOF | RDB + AOF |
| **集群** | Redis Cluster (成熟) | 原生（成熟中）| Redis Cluster | Active Replication |
| **社区/文档** | ⭐⭐⭐⭐⭐ 最丰富 | ⭐⭐ 成长中 | ⭐⭐⭐ | ⭐⭐ |

### 7.2 选型结论

**首选：Redis**

SPMA 场景下的选型分析：

1. **生态成熟度是决定性因素**——Redis 拥有最完善的客户端库（Python redis-py 是事实标准）、最丰富的运维工具（Sentinel/Cluster/可视化面板）、最详尽的文档和社区支持。对于企业生产环境，"出了问题时能快速定位"比"极致性能"更重要
2. **模块生态完整**——RedisJSON、RedisSearch、RedisGraph、RedisBloom、Redis Streams、RedisTimeSeries 等模块可直接复用。SPMA 后续可能需要 Redis Streams 做异步消息、RedisSearch 做轻量全文索引
3. **团队技能匹配**——Redis 是业界最熟悉的内存数据库，运维团队无需额外学习新工具的配置、监控、故障排查流程
4. **SPMA 的数据量在 Redis 舒适区**——Agent 热状态（TTL=5min，万级 keys）+ 热点问答缓存（TTL=1h，十万级 keys），8GB 实例轻松承载，无需 Dragonfly 的内存优势

**为什么不用 DragonflyDB？** DragonflyDB 在吞吐量和内存效率上确实优于 Redis，但其运维生态远不如 Redis 成熟——缺少成熟的集群方案、监控面板较少、社区规模有限。对 SPMA 的数据量级（非海量缓存场景），Redis 的性能完全够用，生态优势远大于性能差异。

**备选：Valkey**

如果企业对 Redis Ltd. 的 RSAL/SSPL 许可证变更（2024年3月）有顾虑，Valkey（Linux Foundation 治理，BSD-3 许可）是安全的替代方案——100% Redis 兼容，可直接替换。

### 7.3 配置建议

| 配置项 | 推荐值 |
|--------|--------|
| 版本 | Redis 7.2+ |
| 内存 | 8 GB（生产），2 GB（开发）|
| 持久化 | RDB 快照（每小时）+ AOF (everysec) |
| 热状态 TTL | 5 min（Agent 循环内状态）|
| 热点缓存 TTL | 1h（问答结果），5min（查询结果）|
| 高可用 | Redis Sentinel (Phase 2+) / Redis Cluster (Phase 4+) |
| 连接池 | redis-py ConnectionPool, max_connections=20 |

## 八、API 网关选型

### 8.1 行业对比

| 维度 | **Apache APISIX** ✅ | Kong | Traefik |
|------|-------------------|------|---------|
| **QPS @ 1000 并发** | **~50,000** | ~38,000 | ~28,000 |
| **P95 延迟 (高负载)** | **~60ms** | ~80ms | ~90ms |
| **配置热加载** | **亚毫秒 (etcd)** | 秒级 (DB 轮询) | 秒级 (watch) |
| **插件数量** | 100+ (全部开源) | 60+ (部分企业版) | ~30 (middleware) |
| **认证 (OIDC/JWT/LDAP)** | ✅ 全部内置开源 | ⚠️ OIDC 需企业版 | Forward Auth 模式 |
| **高级限流 (滑动窗口/令牌桶)** | ✅ 全部开源 | ⚠️ 企业版 | ❌ 基础限流 |
| **AI/LLM 令牌限流** | ✅ (v3.8+) | ⚠️ 新兴 | ❌ |
| **多协议** | HTTP/gRPC/WS/TCP/UDP/MQTT/Dubbo | HTTP/gRPC/WS/TCP | HTTP/gRPC/WS/TCP |
| **管理 Dashboard** | ✅ 内置开源 | ⚠️ 企业版 | ✅ 内置 |
| **License** | Apache 2.0 | Apache 2.0 + Enterprise | MIT |
| **运维复杂度** | 中（需 etcd 集群）| 中（需 PostgreSQL）| 低（K8s CRDs）|

### 8.2 选型结论

**首选：Apache APISIX**

选择 APISIX 的决定性因素：

1. **LLM 令牌级限流**——APISIX 3.8+ 内置了针对 LLM/AI 场景的 token-based rate limiting，可直接对 Haiku/Sonnet API 调用做 token 预算控制，与 SPMA 的 Token 预算管理天然对接
2. **全部核心功能开源**——OIDC、JWT、LDAP、高级限流（滑动窗口）在 APISIX 都是开源内置的，不需要企业版。Kong 的 OIDC 和高级限流需要付费
3. **亚毫秒热加载**——etcd 驱动的配置变更无需重启，这对 Agent feature flags（秒级回退到 pipeline 模式）至关重要
4. **性能最优**——在 1000 并发的代理场景下，APISIX 的 QPS 比 Kong 高约 32%，延迟低约 25%

**备选：Kong**
如果企业已有 Kong 运维经验、或者需要开发者门户（Developer Portal）能力，Kong Enterprise 是不错的选择。

### 8.3 配置要点

| 配置项 | 推荐值 |
|--------|--------|
| etcd 集群 | 3 节点 (HA) |
| 限流策略 | sliding_window + Redis 共享计数 |
| 认证链 | JWT 验证 → OIDC 转发 → API Key 降级 |
| LLM 代理 | APISIX AI 插件 → Haiku/Sonnet API 路由 + token 预算注入 |
| 日志 | HTTP Logger 插件 → ClickHouse (Langfuse 共享) |
| 部署 | K8s DaemonSet 或 Deployment (2 副本) |

```
                        三层可观测架构

|------|----------|--------|-------|

Langfuse 在 Agent 循环追踪、Prompt 版本管理、Token 成本分析三个维度最适合 SPMA 的需求。如果未来深度 RAG 评估需求增长，可叠加 Arize Phoenix 专门做 embedding drift detection。

## 十、重排序模型选型

### 10.1 行业对比

| 模型 | 类型 | ELO (2025) | 延迟 | 成本 | 中文能力 |
|------|------|------------|------|------|---------|
| **BGE-Reranker v2 M3** | 开源 Cross-Encoder | 1327 | ~50ms (GPU) | 零（本地）| ⭐⭐⭐⭐⭐ |
| **Qwen3-Reranker 8B** | 开源 Cross-Encoder | **1473** | ~100ms (GPU) | 零（本地）| ⭐⭐⭐⭐⭐ |
| Cohere Rerank 4 Pro | 闭源 API | 1629 | 614ms | $0.05/M tokens | ⭐⭐⭐⭐ |
| Voyage Rerank 2.5 | 闭源 API | 1544 | 613ms | $0.05/M tokens | ⭐⭐⭐⭐ |
| Cohere Rerank 4 Fast | 闭源 API | 1510 | **447ms** | $0.05/M tokens | ⭐⭐⭐⭐ |

### 10.2 选型结论

**Phase 1-2：不引入 Reranker（RRF 等权融合足够）**

设计文档明确指出 Phase 1 不加权，等权 RRF 是最稳健的起步选择。权重应由线上数据驱动确定。

**Phase 3：BGE-Reranker v2 M3 或 Qwen3-Reranker 8B（本地）**

当 RRF 权重优化空间耗尽后，引入本地 Reranker 对 RRF Top-20 做精排：
- BGE-Reranker v2 M3：中英双语深度优化，与 BGE-M3 嵌入模型配套
- Qwen3-Reranker 8B：ELO 更高（1473 vs 1327），但模型较大，需额外 GPU 资源

**为什么不选商业 API Reranker？** Cohere Rerank 4 Pro 精度最高，但 614ms 延迟对于 10s 硬上限的总预算来说太贵。本地 Reranker 在 50ms 内完成，且数据不出境。

├─ 是 → ✅ LangChain 1.x create_agent() (一行代码，自动 ReAct 循环)
│      └─ 是否需要 Middleware (脱敏/HITL/摘要)？
│           ├─ 是 → 挂载对应 Middleware 即可
│           └─ 否 → 直接使用
│
└─ 否 → 是否需要复杂编排（多Agent并行/自定义状态机/收敛契约）？
       ├─ 是 → ✅ LangChain 1.x + LangGraph StateGraph
       │      ├─ 多Agent并行派发 → LangGraph Send API
       │      ├─ 长任务断点恢复 → LangGraph Checkpointer
       │      └─ 自定义循环逻辑 → StateGraph 条件边 + 节点
       │
       └─ 否 → 是否为快速原型验证？
              ├─ 是 → CrewAI (2-4h 可出 Demo)
              └─ 否 → ✅ LangChain 1.x create_agent (生产首选)
```

### 17.2 向量数据库选型决策树

```
已有PostgreSQL运维能力？
├─ 是 → 向量规模预估 < 5000万？
│      ├─ 是 → ✅ PGVector (零新服务, ACID, SQL JOIN)
│      └─ 否 → 需要极低延迟 (<20ms P99)?
│             ├─ 是 → Qdrant
│             └─ 否 → Milvus (十亿级以上)
└─ 否 → 向量+关键词混合检索优先？
       ├─ 是 → Weaviate
       └─ 否 → 延迟敏感 → Qdrant / 规模优先 → Milvus
```

### 17.3 LLM 选型决策树

```
任务类型？
├─ Agent完备度判断/语义验证 → ✅ Claude Haiku (快速, 便宜, 中文强)
├─ 意图分类/实体抽取 → ✅ Claude Sonnet (高质量分类)
├─ 最终回答生成 → ✅ Claude Sonnet (高质量中文)
├─ SQL生成 → ✅ Claude Sonnet (复杂SQL推理)
└─ 以上不可用/降级 → ✅ Qwen3-8B (本地vLLM, 零API成本)
```

### 17.4 全文检索引擎选型决策树

```
文档规模？
├─ < 100万片段, Phase 1-2 → ✅ PostgreSQL tsvector + zhparser
├─ 100万-1000万, Phase 3 → ✅ Elasticsearch (IK分词, kNN)
│   └─ 是否需要开发者门户？ → Kong Enterprise
├─ 追求极致简单，< 1000万文档 → Meilisearch (零调优, 好默认值)
└─ 追求极低延迟，< 1000万文档 → Typesense (<5ms P50)
```

## 二十、风险与缓解

### 20.1 关键风险矩阵

| 风险 | 等级 | 缓解措施 | 监控指标 |
|------|------|---------|---------|
| LangChain 1.x + LangGraph 学习曲线 | 🟡 中 | Phase 0 即开始团队培训；create_agent 高层 API 降低入门门槛，复杂编排逐 Agent 下沉到 StateGraph | 开发进度 |
| PGVector 性能瓶颈 | 🟡 中 | 预留 Qdrant 迁移接口；监控 P99 延迟和向量规模增长趋势 | `pgvector_p99_latency` |
| BGE-M3 中→英代码符号映射质量 | 🟡 中 | 同义词映射表人工种子 + 自动发现闭环；Phase 2 前完成评估 | 搜索词构造命中率 |
| LLM API 不可用 | 🟡 中 | Qwen3-8B 本地降级；多级降级 L0-L4 | `llm_availability` |
| Token 成本爆炸 | 🟡 中 | Token 预算管理 + Haiku 分担大部分调用 + Langfuse 成本热力图 | `token_cost_per_query` |
| Agent 延迟不可预测 | 🟡 中 | 收敛契约 + 10s 硬上限 + P50 目标保障 | `agent_rounds_p99` |
| 虚假信心（Agent 说够了但不够）| 🟡 中 | Agent Eval Dataset + 虚假信心率指标 + Plan B (纯确定性收敛) | `agent_false_confidence_rate` |
| 嵌入模型推理GPU不足 | 🟢 低 | 量化 (AWQ INT4) + 批处理 + GPU 扩容 | GPU 利用率 |
| BGE-Reranker 精度不如商业模型 | 🟢 低 | RRF 等权长期可用（优于预期）；Qwen3-Reranker 备选 | NDCG@10 离线评估 |

### 20.2 不静默失败原则

任何降级路径被触发时，必须在返回给用户的回答末尾附加透明标注，例如：
- 检索降级：`ℹ️ 本次查询使用了全文语义搜索，未识别到精确需求ID，结果可能不够精准`
- Agent 超时：`⏱️ 查询超时，以下结果可能不完整`
- LLM 降级：后台自动切换，不标注（用户无感知）
