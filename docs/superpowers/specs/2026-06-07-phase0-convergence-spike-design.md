## 一、设计范围与目标

### 1.1 核心问题

**验证 LLM 能否可靠判断检索结果"够了"**——这是整个 5 Agent 架构的基石。

### 1.2 设计决策摘要

| 决策维度 | 选择 |
|---------|------|
| 检索模拟 | 简易 BM25 + embedding 混合检索管线，独立于项目 src/ |
| 标注方式 | 双人全量标注 + IAA ≥ 0.8（Cohen's Kappa）+ 分歧第三人裁决 |
| 多轮构造 | 规则驱动的 Agent 循环模拟器，第 1 轮真实检索 + 后续轮次逐步注入 golden 结果 |
| Prompt 工程 | 三种方案（简洁/结构化/多维度评分）在 5 折 CV 下对比，选优后 FP 案例驱动的定向改进 |
| 评估方法 | 5 折交叉验证 + 分层分析 + FP 可修复性分类 |
| 通关标准 | 精确率 ≥ 80%（LLM 说够了时至少 80% 概率真的够了） |

### 1.3 Go/No-Go 决策矩阵

| 结果 | 决策 | 后续行动 |
|------|------|---------|
| 精确率 ≥ 85% | ✅ **Go** — 强烈信心进入 Phase 1 | 开始实现 Agent 循环 + LLM 完备度判断 |
| 精确率 80%-85% | ✅ **Go** — 有条件进入 Phase 1 | Agent 循环中增加确定性收敛的权重，LLM 判断作为辅助 |
| 精确率 70%-80% | ⚠️ **Risk** — 需评审 | 分析 FP 案例是否可通过 Prompt 改进修复；若可修复则重新评估 |
| 精确率 < 70% | ❌ **No-Go** — 启用 Plan B | 放弃 LLM 完备度判断，改用纯确定性收敛 + 轮次上限 |

## 三、标注规范："信息充足"判定 Rubric

### 3.1 核心判定原则

> **"有了这些检索结果，一个熟悉企业业务的人能否回答用户的问题？"**

不要求"回答得完美"或"覆盖所有细节"，而是"能回答核心问题，不会因为关键信息缺失而答错或答不了"。

### 3.2 判定 Rubric

| 等级 | 标签 | 定义 | 示例 |
|------|------|------|------|
| S3 | **完全充足** | 所有子问题都有直接覆盖，关键实体全部命中，结果之间有足够多样性（不重复） | 问"登录模块 PRD 改了啥、影响哪些代码文件" → Doc 搜到 PRD 全文 + diff，Code 搜到所有关联文件 |
| S2 | **基本充足** | 核心问题能回答，但次要细节可能缺失或部分结果不够精确 | 搜到了 PRD 和主要代码文件，但少了一个边缘依赖文件——不影响回答核心问题 |
| S1 | **勉强充足** | 核心问题能大致回答但信息有 gap，回答会不完整或需要推断 | 只有 PRD 没搜到代码，但从 PRD 描述可以推断出可能影响的模块 |
| I1 | **轻度不足** | 缺少某个关键实体或子问题的覆盖，直接回答会出错 | 搜到了相关文档但缺少核心代码文件，无法给出代码层面的答案 |
| I2 | **明显不足** | 多个子问题无覆盖，或核心实体大面积缺失 | 跨源查询只搜到了 Doc，Code 和 SQL 表都没搜到 |
| I3 | **完全不相关** | 检索结果跟用户问题基本无关 | 问"用户登录"搜到了"会议室预订系统" |

### 3.3 判定映射

- **"够了" (sufficient)**：S3、S2、S1 → golden label = `sufficient`
- **"不够" (insufficient)**：I1、I2、I3 → golden label = `insufficient`

S1（勉强充足）归为"够了"——**宁可放行、不少搜**，与"精确率优先、召回率不做硬性要求"的策略一致。

### 3.4 标注流程

```
标注者拿到 → query + 本轮检索结果列表（含文档/代码/SQL表）
          ↓
逐条检查检索结果是否与 query 相关
          ↓
判断每个子问题是否至少有一条相关结果覆盖
          ↓
按 Rubric 给出 S3/S2/S1/I1/I2/I3 标签
          ↓
标签映射为 sufficient / insufficient
```

### 3.5 多轮标注的特殊处理

每条 query 有 2-3 轮检索结果（每轮递增），标注者对**每一轮**独立判断。不做跨轮比较（不要让标注者知道"这是第几轮"），只基于当前轮的结果判定。

## 四、检索模拟 & 多轮 Agent 循环模拟器

### 4.1 数据源采样

从项目三类数据源各取子集构建 Spike 本地索引：

| 数据源 | 采样规模 | 格式 |
|--------|---------|------|
| Confluence 文档 | ~200 页（覆盖不同模块） | 每页 split 为 chunks，带 doc_id + 标题 |
| Git 代码仓库 | ~300 个文件（核心模块） | 每文件带文件路径 + 函数/类列表 |
| SQL Schema | 完整的 table DDL + 字段注释 | 用作 SQL Agent 的 schema 检索 |

全部导出为 JSON 文件放在 `spike/data/corpus/` 下，不需要连真实数据库。

### 4.2 简易检索管线

```
spike/retrieval/
├── indexer.py       # BM25 索引构建 + embedding 向量生成
├── searcher.py      # 混合检索：BM25(α=0.3) + cosine similarity(β=0.7)
├── corpus_loader.py # 从 JSON 加载三种语料
└── config.yaml      # 索引路径、模型名
```

**模型选择：** 默认 `all-MiniLM-L6-v2`（本地 CPU 可跑、384 维）。如中文效果差，备选 `BAAI/bge-small-zh-v1.5`。

### 4.3 多轮 Agent 循环模拟器

不写真实 Agent 代码，而是用一个**规则驱动的轮次模拟器**：

```python
# spike/simulation/multi_round_simulator.py

class MultiRoundSimulator:
    """
    模拟 Doc/Code/SQL Agent 的多轮检索行为。

    输入: query + golden_results(完整标注)
    输出: [
        {round: 1, results: [...], golden_label: "insufficient"},
        {round: 2, results: [...], golden_label: "insufficient"},
        {round: 3, results: [...], golden_label: "sufficient"},
    ]
    """
```

**模拟策略（按 Agent 类型）：**

| Agent | 第 1 轮 | 第 2 轮 | 第 3 轮 |
|-------|---------|---------|---------|
| **Doc** | BM25 + embedding 混合检索，取 Top-5 | 从 golden docs 中随机补充 2-3 条未命中项（模拟"线索扩展重搜"） | 再补充剩余 golden docs（模拟"够了"） |
| **Code** | ripgrep 风格关键词匹配，取 Top-3 | 从 golden code 中补充调用链相关文件 | 补充剩余 golden code |
| **SQL** | Schema RAG 检索相关表，生成 1 条 SQL | 修正 SQL + 补充遗漏表 | 最终 SQL + 完整表集合 |

核心思路：**第 1 轮用真实检索结果（体现真实分布），后续轮次逐步注入 golden 结果（模拟 Agent 多轮改善）**。这样评估的分布偏差在可接受范围内。

### 4.4 轮次构造规则（非 golden label）

多轮模拟器用覆盖率决定**轮次构造**（在第几轮停止注入 golden 结果，使该轮达到"够了"状态）：

```
第 N 轮覆盖率 = 本轮已注入的 golden items / 全部 golden items
覆盖率 ≥ 80% → 该轮标记为"目标充足轮"，模拟器停止注入
覆盖率 < 80%  → 继续注入更多 golden items 到下一轮
```

80% 阈值基于 Synthesis Agent 的收敛条件："引用覆盖率 ≥ 80% claim"。

**注意：** 此覆盖率仅用于驱动模拟器构造各轮检索结果，**不作为 golden label**。每轮的 golden label 始终由双人标注者按 Rubric（第三节）人工判定。这确保了评估的独立性——模拟器不知道标注者认为"够了"的标准，标注者也不知道模拟器的覆盖率阈值。

## 五、Prompt 工程实验设计

### 5.1 三种 Prompt 方案

**Prompt A：简洁判断（Baseline）**

```
你是检索质量评估器。判断以下检索结果是否足以回答用户问题。

用户问题: {query}

检索结果:
{results_json}

只需回答: 够了 / 不够
```

- 输入 tokens：~800 | 预期延迟：~200ms
- 风险：LLM 没有结构化推理过程，容易受表面数量影响

**Prompt B：结构化逐项检查**

在 PRD 模板基础上加入 3-shot 示例（1 个 sufficient + 1 个 insufficient + 1 个边缘 case）和具体的检查维度。输出 JSON 含逐项检查结果 + 最终 verdict。

- 输入 tokens：~1500 | 预期延迟：~350ms
- 风险：维度定义可能不够泛化，特定 query 类型覆盖率不足

**Prompt C：多维度评分 + 阈值判定**

每维度 0-1 评分，加权总分 ≥ 0.7 → sufficient。

| 维度 | 权重 | 说明 |
|------|------|------|
| 实体覆盖度 | 0.35 | 关键实体（需求ID、文件名、表名）命中率 |
| 数量充足度 | 0.20 | 结果数量/多样性是否达标 |
| 语义覆盖度 | 0.30 | 每个子问题是否至少有一条相关结果 |
| 结果新鲜度 | 0.15 | 时间相关查询的结果时效性 |

- 输入 tokens：~2000 | 预期延迟：~500ms
- 风险：权重和阈值需要校准，可能过拟合

### 5.2 实验流程

```
                   5折 CV ────────────────────────┐
                                                   │
  Fold 1: train(80条) ─→ 三套Prompt各跑 → 记精确率  │
  Fold 2: train(80条) ─→ 三套Prompt各跑 → 记精确率  │
  Fold 3: train(80条) ─→ 三套Prompt各跑 → 记精确率  │
  Fold 4: train(80条) ─→ 三套Prompt各跑 → 记精确率  │
  Fold 5: train(80条) ─→ 三套Prompt各跑 → 记精确率  │
                                                   │
                    └──→ 每套Prompt的5折平均精确率+方差
                              ↓
                        选最优Prompt
                              ↓
                    FP案例分析(仅在train folds上)
                              ↓
                    针对性改进Prompt版本
                              ↓
                    最终5折CV评估(test folds)
```

**关键规则：**
- Prompt 调优**只能看 train folds 的 FP 案例**，绝不能看 test folds
- 改进后的 Prompt 版本视为全新 Prompt，重新跑一轮 5 折 CV
- 最终评估报告引用的是**改进后 Prompt 的 test folds 指标**

### 5.3 FP 案例分析

每条 FP（LLM 说够了但不够），记录：

1. **query + 本轮检索结果**：LLM 看到了什么
2. **LLM 输出**：它的判断理由
3. **为什么实际不够**：缺失了什么关键信息
4. **FP 类型标注**：实体遗漏 / 数量幻觉 / 语义理解错误 / 其他
5. **可修复性判断**：Prompt 调整能修复？还是结构性缺陷？

如果 ≥ 60% 的 FP 是可修复的 → 在 Prompt 层面加规则/示例解决。如果 ≥ 40% 的 FP 是结构性的 → 这是 Plan B 的论据。

## 六、评估方法论 & 统计分析

### 6.1 核心指标

```
                           实际够了    实际不够
LLM 判断"够了"               TP          FP
LLM 判断"不够"               FN          TN
```

| 指标 | 公式 | 目标 | 说明 |
|------|------|------|------|
| **精确率 (Precision)** | TP / (TP + FP) | **≥ 80%** | 通关标准——LLM 说够了时真的够了 |
| 召回率 (Recall) | TP / (TP + FN) | 不做硬性要求 | 宁可多搜也不少搜——多搜一轮代价低 |
| F1 | 2×P×R/(P+R) | 参考值 | 综合指标 |
| 特异度 (Specificity) | TN / (TN + FP) | 参考值 | LLM 识别"不够"的能力 |

**为什么精确率是通关标准？** FP 代价高——LLM 说够了，Agent 停止搜索，用户得到不完整的答案；FN 代价低——LLM 说不够，Agent 多搜一轮，多花 500ms。

### 6.2 5 折 CV 统计报告

```
最终报告指标格式:
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│          │  Fold 1  │  Fold 2  │  Fold 3  │  Fold 4  │  Fold 5  │  平均±标准差 │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┼────────────┤
│ 精确率    │  0.83    │  0.79    │  0.85    │  0.81    │  0.82    │ 0.820±0.021│
│ 召回率    │   ...    │   ...    │   ...    │   ...    │   ...    │   ...±...  │
│ 判断点数量 │   58     │   62     │   55     │   60     │   57     │ 292 total  │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┴────────────┘
```

每个 fold 的判断点是 query×round 的聚合（100 条 × 2-3 轮 ≈ 250-300 个判断点）。

### 6.3 分层分析

| 分层维度 | 分组 | 分析指标 |
|---------|------|---------|
| 按 query_type | 单源Doc / 单源Code / 单源SQL / 跨源双源 / 跨源三源 | 各组的精确率、FP 数量 |
| 按 Agent 类型 | Doc / Code / SQL / 跨源 | 哪个 Agent 的完备度判断最不准 |
| 按轮次 | Round 1 / Round 2 / Round 3 | 哪一轮 LLM 最容易误判 |
| 按查询长度 | 短（<15字）/ 中（15-50字）/ 长（>50字） | 查询越短越模糊，精确率是否下降 |
| 按结果数量 | 结果 ≤2 / 3-5 / 6-10 / >10 | 结果越多 LLM 是否越倾向说够了 |
| 按覆盖率区间 | 实际覆盖率 <40% / 40-60% / 60-80% / >80% | 在"临界区间"(60-80%)，LLM 判断最困难？ |

### 6.4 FP 可修复性分类

```
所有FP案例 ─→ 分类:
  ├── A类: 实体遗漏型    (LLM 忽略了某个关键实体缺失)
  ├── B类: 数量幻觉型    (LLM 被结果数量迷惑，不检查质量)
  ├── C类: 语义理解错误  (LLM 误解了 query 或结果内容)
  ├── D类: 跨源关联失败  (LLM 没发现跨源信息之间的缺口)
  └── E类: 其他          (标注错误、极端 case 等)

可修: A/B（Prompt 加检查项/示例可解决）
结构性: C/D/E（LLM 能力瓶颈，需要架构层面解决）
```

### 6.5 IAA 计算

使用 **Cohen's Kappa**（而非简单一致率），因为 sufficient/insufficient 可能存在类别不平衡：

```
Kappa = (Po - Pe) / (1 - Pe)
Po = 观察一致率, Pe = 随机一致率

目标: κ ≥ 0.8
可接受: κ ≥ 0.6（需改进 Rubric 后重标）
不合格: κ < 0.6（重新设计 Rubric）
```

### 7.2 人工覆盖边界——分类矩阵

20 条人工 query 按以下矩阵构造，确保没有类型缺口：

| | 短查询 (<15字) | 中查询 (15-50字) | 长查询 (>50字) |
|---|---|---|---|
| **单源 Doc** | "PRD 文档搜索" | "用户登录模块的需求变更历史" | ... |
| **单源 Code** | "auth.py" | "OAuth 2.0 实现的所在文件" | ... |
| **单源 SQL** | "用户表结构" | "近一周注册用户的转化率" | ... |
| **跨源双源** | ... | ... | ... |
| **跨源三源** | "订单退款" | "退款功能的 PRD 改了啥，影响了哪些代码和表？" | （10字以内的三源查询几乎不存在） |

每格至少 1 条，总计 ≥ 20 条。确保覆盖精确 ID 查询、模糊自然语言、时间范围、否定/反事实等边界类型。

### 7.3 标注流程

```
100 条 query 审核通过
       ↓
简易检索管线 → 对每条 query 跑混合检索 → 生成结果池(每query约15条候选)
       ↓
多轮模拟器 → 构造每轮结果(2-3轮) → ~280个判断点
       ↓
┌─────────────────────────────────────────────┐
│ 标注者A: 逐判断点标注 S3/S2/S1/I1/I2/I3    │
│ 标注者B: 同上(全量标注)                      │
│ 每轮独立判断，不知晓是第几轮                   │
└─────────────────────────────────────────────┘
       ↓
20 条 query 先标注 → 计算 IAA
       ↓
IAA ≥ 0.6 但 < 0.8? → 重新校准 Rubric → 继续
IAA < 0.6? → 重新设计 Rubric → 重新标注
IAA ≥ 0.8? → 继续标注剩余 80 条
       ↓
分歧点(标签不同) → 第三人裁决 → golden label
       ↓
golden 评测数据集: spike/data/golden_eval_dataset.json
```

## 十一、风险与缓解

| 风险 | 概率 | 缓解 |
|------|------|------|
| 标注质量不达标（IAA < 0.6） | 中 | 20 条先导标注检测，Rubric 迭代至达标后再继续；预留 1 周 buffer |
| Haiku API 不可用或延迟过高 | 低 | 预留 Qwen3-8B 本地替代方案（结果不纳入正式评估，仅验证管线可跑） |
| Prompt 工程陷入局部最优或过拟合 train folds | 中 | 5 折 CV 框架保证 test fold 独立；最终指标以 test folds 为准 |
| 简易检索管线与真实管线差距过大 | 中 | 在评估报告中显式记录"模拟与真实偏差"，不将此偏差隐蔽 |
| 双人标注工作量超出预算 | 中 | 预计 100 条 × 2-3 轮 ≈ 280 个判断点，每点 1-2 分钟，双人约 14-19 小时可完成 |

## 十二、与 PRD 的差异说明

| 维度 | PRD 原方案 | 本设计（方案 B） | 差异原因 |
|------|-----------|-----------------|---------|
| 工期 | 0.5 人月 (2周) | ~3 周 | 增加双人标注、5折CV、完整消融实验 |
| 检索模拟 | 未详述 | 独立 BM25+embedding 管线 | 需产生真实分布的结果 |
| 标注 | 未明确 | 双人全量 + IAA + 分歧裁决 | 保证 golden label 质量 |
| 多轮 | 仅第1轮 | 每条 2-3 轮，半自动模拟 | 更贴近 Agent 循环实际场景 |
| 评估 | 简单 train/test | 5折CV + 分层 + 置信区间 | 统计可信度 |
| Prompt 对比 | 提及 3 方案 | 5折CV框架下严格对比 | 防过拟合 |
