## Problem Statement

产品经理和开发工程师在企业内部面临严重的信息碎片化问题。三种核心数据来源——PRD 产品文档、开发代码、SQL 数据库——分散在不同的系统和工具中（Confluence/Wiki、IDE、数据库客户端），每次查询需要切换系统、手动搜索、人脑关联。这种"三系统切换"的模式每天浪费大量时间，导致：

- **产品经理**：查不到历史需求的完整上下文，做决策缺乏数据支撑
- **开发工程师**：故障定位时无法快速追溯代码到需求，查数据验证问题需要找 DBA 或手写 SQL
- **两拨人的共同痛点**：无法用自然语言直接查询业务数据

## Demand Evidence

- 具体的团队同事明确投诉过这个问题——有具体的人和场景，不是泛泛的"可能有用"
- 现状是多系统切换 + 人工搜索，每天切换 5-10+ 次
- 两拨用户群体都有明确的使用场景：PM 的需求溯源场景、开发的故障定位场景、共同的数据查询场景

## Status Quo

当前的"竞品"是碎片化的工具组合：

| 用户 | 现状 | 代价 |
|------|------|------|
| 产品经理 | Confluence/Wiki 搜索 PRD 文档 | 找不到历史版本、无法对比、需要人脑关联需求变更 |
| 开发工程师 | IDE grep 搜索代码 + 手动追踪需求 | 故障定位慢、新人上手周期长（依赖老员工口传） |
| 双方共同 | 找 DBA 或手写 SQL 查询数据 | 瓶颈在 DBA 带宽、非技术人员完全无法自助 |

信息获取依赖人际网络——新人入职要几个月才能独立工作。这是企业知识管理的典型失效模式。

## Target User & Narrowest Wedge

**核心用户画像：**
- 产品经理：需要需求溯源能力——某个功能为什么这么做？历史版本怎么演化的？这个改动影响了哪些下游？
- 开发工程师：需要快速定位能力——这个 bug 跟哪个需求相关？涉及哪些代码模块？影响了哪些数据表？

**Wedge：** 三源全覆盖 + 企业级工程化——用户明确要求不是 MVP 原型，而是从第一天就按生产标准设计的完整系统。对 intrapreneurship 场景来说，这是合理的：内部工具如果不可靠，用户不会用第二次。

## Constraints

- 企业内网部署，敏感数据不出企业边界。LLM 调用必须经过数据脱敏层（脱敏后可通过外网 API），或使用本地部署模型
- 三种数据源差异极大（自然语言文档、结构化代码、关系型数据库），不能用一套检索策略覆盖
- 内部用户对响应延迟敏感，整体硬上限 10s
- 需要支持中文内容（PRD 文档、自然语言查询、中文代码注释）
- 系统可靠性决定用户信任——一次查询失败可能导致用户放弃使用

## Premises

1. **三种数据源需要三种不同的检索策略** — PRD 文档用语义搜索+关键词混合检索，代码用 ripgrep 实时搜索，SQL 用 Schema 感知的 RAG 检索。不能用一套策略打天下。

2. **5 独立 Agent 架构** — Supervisor Agent 编排调度，Doc Agent / Code Agent / SQL Agent 各自独立检索循环，Synthesis Agent 审计融合。每个 Agent 内部具备多轮自主推理能力，通过收敛契约控制循环次数。详见 [SPMA-design-07](SPMA-design-07-agent-architecture.md)。

3. **收敛契约驱动** — 每个 Agent 有明确的最大轮数、收敛条件和超时限制。Supervisor 通过质量函数评估 Worker 结果，决定是否需要重调度。这是整个架构的基石。

4. **四级降级保障** — 全功能 → LLM 降级（切换备用模型）→ Agent 降级为单轮 pipeline → 缓存兜底（热点问答）→ 静态 FAQ 兜底。每一级自动触发、自动恢复。每个 Agent 有独立 feature flag，可秒级回退。

5. **Text-to-SQL 必须包含 SQL Guard 层** — 语法校验 + DDL/DML 拦截 + 只读副本执行 + 语义验证循环。生产环境中不允许任何写操作风险。

6. **三源平等支柱** — PRD 文档、代码、SQL 是三个同等重要的检索入口。跨源溯源（"这个需求影响了哪些代码和哪些表？"）是最核心的使用场景。

## Approaches Considered

### Approach A: 5 Agent 架构 ✅ 选中
**Summary:** Supervisor Agent 编排层理解意图 → Send API 并行派发到 3 个 Worker Agent（Doc/Code/SQL 各自独立检索循环）→ Synthesis Agent 审计融合。每个 Agent 内部多轮自主推理，收敛契约控制循环次数。
**Effort:** 人：5-7.5 人·月（2-3 人并行约 3-5 自然月）；AI 辅助可加速代码生成但无法缩短架构决策和数据源对接时间
**Risk:** 中（Agent 编排的复杂性需要精心设计收敛契约和 fallback）
**Pros:**
- 每种数据源用最优检索策略，不妥协
- 天然支持跨源多跳查询（核心价值场景）
- 收敛契约确保每个 Agent 不会无限循环
- 每个 Agent 独立 feature flag，可秒级回退到 pipeline 模式
- 易于扩展更多数据源
**Cons:**
- Agentic 编排增加延迟和工程复杂度
- 需要精心设计收敛条件、超时处理和回退路径
- 调试和可观测性要求高于简单管道

### Approach B: 联邦并行检索 RAG
**Summary:** 不做智能路由——每个查询同时发到三个专门化检索管道，各自独立返回，用 RRF（Reciprocal Rank Fusion）融合排序后统一展示。
**Effort:** 人：~2-3 个月 / AI 辅助：~1-2 周
**Risk:** 低
**Pros:**
- 延迟更低（并行执行，无路由决策开销）
- 架构简单，调试容易
- 每条管道独立优化、独立故障隔离
**Cons:**
- 不支持多跳跨源推理（"先查文档再决定查哪个表"）
- 简单查询也要跑三条管道，资源浪费
- 无法动态调整检索策略

### Approach C: 知识图谱 + RAG 混合
**Summary:** 构建统一知识图谱——PRD 需求节点、代码模块节点、数据库表节点用关系边连接。RAG 做初始检索，图遍历做关系追踪。
**Effort:** 人：~4-6 个月 / AI 辅助：~3-4 周
**Risk:** 高
**Pros:**
- 最强的跨源溯源能力（确定性关系查询，不会幻觉）
- 对"影响分析"类问题最优
- 图结构天然支持版本演化和时间线查询
**Cons:**
- 构建和维护知识图谱成本极高
- 需要持续同步三种数据源的变化（CDC 链路）
- 冷启动问题（初始图谱构建工作量大）

### 5 Agent 概览

| Agent | 类型 | 循环模型 | 收敛条件 | 超时 |
|-------|------|---------|---------|------|
| **Supervisor Agent** | 编排 Agent | 分发 → 收集 → 质量评估 → 不够 → 调整参数重分发 | 所有Worker评分≥0.6 OR 重调度2次无改善 | 5s |
| **Doc Agent** | 检索 Agent | BM25+向量混合检索 → 完备度判断 → 不够 → 线索扩展重搜 | 结果≥5条 AND (req_ids精确匹配 OR LLM判断"信息充足") | 2s |
| **Code Agent** | 检索 Agent | ripgrep → 完备度判断 → 不够 → 调用链展开重搜 | 结果≥3条 AND (调用链深度≤2层 OR 第3轮无新增文件) | 2s |
| **SQL Agent** | 执行 Agent | Schema RAG → LLM生成SQL → Guard → 执行 → 语义验证 → 不够 → 重生成 | SQL执行成功 AND 行数∈[1,10000] AND 通过语义验证 | 3s |
| **Synthesis Agent** | 审计 Agent | RRF融合 → LLM生成初稿 → 引用完整性检查 → 不够 → 修正 | 引用覆盖率≥80% claim AND 无跨源矛盾 | 2s |

**整体硬上限：10s** → 所有 Agent 强制停止 → 返回部分结果 + "⏱️ 查询超时，以下结果可能不完整"。

> 详细架构设计（收敛契约、质量函数、状态管理、Agent交互协议等）见 [5独立Agent架构设计](SPMA-design-07-agent-architecture.md)。

## 跨源多跳查询 Trace 示例（Agent 架构版）

用户问题：**"用户登录模块的 PRD 需求改了哪些内容？影响了哪些代码文件和数据库表？"**

```
T=0.0s   Supervisor Agent — Round 1: Query Understanding
           ├─ 意图分类: cross-source (doc + code + sql)
           ├─ 实体抽取: {module: "用户登录", actions: ["PRD变更", "代码影响", "数据库影响"]}
           ├─ Send API 并行派发 → Doc Agent, Code Agent, SQL Agent
           └─ [延迟预算: 500ms]

T=0.5s   Worker Agents — 并行执行各自检索循环
           ├─ Doc Agent (≤3轮, 2s超时):
           │   Round 1: BM25+向量混合检索 → Top-K 文档片段
           │   完备度判断: req_ids命中 → 收敛 ✓
           │   返回: {req_ids: ["REQ-2024-0187"], changes: [{desc: "增加OAuth2.0登录"}, ...]}
           │
           ├─ Code Agent (≤3轮, 2s超时):
           │   Round 1: ripgrep 搜索 "REQ-2024-0187" + "oauth" + "login"
           │   完备度判断: 结果≥3条 + 调用链深度≤2层 → 收敛 ✓
           │   返回: [{file: "src/auth/oauth.py", funcs: ["login_oauth", "token_refresh"]}, ...]
           │
           └─ SQL Agent (≤5轮, 3s超时):
               Round 1: Schema RAG → LLM生成SQL → Guard通过 → 执行成功 → 语义验证通过 → 收敛 ✓
               返回: [{table: "users", cols: [...]}, {table: "user_sessions", cols: [...]}]

T=3.0s   Supervisor Agent — 收集 Worker 结果
           ├─ 质量评估: Doc 0.85, Code 0.78, SQL 0.82 → 全部 ≥ 0.6 ✓
           └─ 收敛，进入 Synthesis

T=3.2s   Synthesis Agent (≤2轮, 2s超时)
           Round 1: RRF融合 + LLM生成初稿
           自检: 引用覆盖率 ≥ 80% → 收敛 ✓
           返回: 结构化回答（含引用标注: [PRD §3.2], [src/auth/oauth.py:42], [users 表]）

总延迟: ~4.2s (P50 < 5s ✓)
最坏情况（满轮+重调度）: ~8.0s (< 10s 硬上限 ✓)
```

> 最坏情况和典型情况的完整时序推演见 [SPMA-design-07 第二节](SPMA-design-07-agent-architecture.md#二收敛契约)。

## 阶段性交付路线图

| Phase | 交付内容 | 用户价值 | 时间（人·月） |
|-------|---------|---------|------------|
| **Phase 0** | **收敛判断 Spike** — 100条 query 验证 LLM 完备度判断精确率 | 架构基石验证，精确率 ≥ 80% 才进入 Phase 1 | 0.5 月 |
| **Phase 1** | SQL Agent（Text-to-SQL + SQL Guard + 语义验证循环）+ 进程内存状态 | PM 和开发都能用自然语言查数据 | 1.5-2 月 |
| **Phase 2** | Doc Agent + Synthesis Agent + Redis 热状态 | PM 可以检索和对比历史 PRD 文档；跨源结果融合审计 | +1-1.5 月 |
| **Phase 3** | Supervisor Agent + Code Agent + Postgres 冷 trace + Agent Dashboard | 开发可以追溯需求→代码→数据的影响链；多轮编排上线 | +1.5-2 月 |
| **Phase 4** | 完整降级 L0-L4 + 熔断器 + K8s 蓝绿部署 + 混沌工程演练 | 企业级可靠性达标（99.9%） | +1-2 月 |
| **Phase 5+** | Cognitive Layer（主动感知 + 跨源推演 + 用户记忆）| 主动式知识发现 | TBD |

**总时间线: 5-7.5 人·月**（2-3 人团队并行约 3-5 个自然月）

每个 Phase 独立交付价值、独立回滚、独立验收。

## Success Criteria

- **检索质量：** Recall@10 ≥ 0.85（跨三种数据源；Ground Truth 来自 Shadowing 收集的 50 条真实用户问题，由领域专家标注正确答案）
- **SQL 准确率：** Text-to-SQL 端到端 Execution Accuracy ≥ 80%（含语义验证循环；评估集：50 条真实业务问题 + 对应正确 SQL）
- **Agent 收敛质量：**
  - 收敛判断 LLM 精确率 ≥ 80%（Phase 0 Spike 验证）
  - 虚假信心率（Agent 说够了但不够）< 15%
  - Agent 早停率（收敛过严）< 30%
- **响应延迟：**
  - 单源查询 P50 < 3s, P95 < 6s, P99 < 8s
  - 跨源查询 P50 < 6s, P95 < 12s, P99 < 15s
  - 整体硬上限 10s（强制中断）
- **可用性：** 系统可用性 ≥ 99.9%（含降级路径；月度统计）
- **Agent 降级率：** < 10%（Redis/LLM 不可用导致的 pipeline 降级）
- **用户满意度：** 内部用户 NPS ≥ 30（上线后 3 个月收集）
- **知识新鲜度：** 文档/代码变更到可检索 < 5 分钟（通过 Git Webhook / CI 触发增量索引）；数据库 Schema 变更 < 10 分钟（定时轮询 + 手动触发刷新）

## Open Questions

1. **PRD 文档存储位置和格式？** Confluence、飞书文档、本地 Markdown？是富文本还是纯文本？不同来源的解析器选择不同。
2. **代码仓库规模和语言？** 单仓还是多仓？主要语言（Python/Java/Go/JS）？代码 commit 中是否有关联需求 ID（如 `[REQ-1234]`）的元数据？这决定了 Code Agent 的 ripgrep 策略和跨源关联的可行性。
3. **SQL 数据库类型和规模？** PostgreSQL、MySQL、还是多种？表的数量级（几十 vs 几千张表？）、是否有 query log 可作为 few-shot 示例来源？
4. **权限模型（v2）？** 是否需要按用户角色限制可查询的数据范围？（PM 不能查薪资表、开发不能查客户隐私数据等）v1 先做全员统一权限，v2 引入 RBAC。
5. **中文→英文代码标识符映射质量？** Code Agent 不依赖代码 embedding，而是通过同义词映射表 + LLM 辅助翻译将中文查询转为英文标识符 grep。需要评估这套映射机制覆盖了多少真实查询场景。Phase 3 前需完成此项评估。
6. **收敛判断 LLM 精确率是否达标？** 这是 Phase 0 的 gating item——如果 LLM 无法可靠判断"找够了"，将回退到纯确定性收敛 + 轮次上限（Plan B）。

## Dependencies

- **数据源接入：** Confluence/Wiki API、Git 仓库访问权限、数据库只读副本
- **LLM 服务：** Claude API（Haiku 用于完备度判断 + Sonnet 用于生成/分类） + 本地 Qwen3-8B（vLLM 部署，API 不可用时的兜底）
- **基础设施：** K8s 集群、PostgreSQL + pgvector、Redis（Agent 热状态存储）
- **Agent 框架：** LangGraph（StateGraph + Send API + Checkpointer）
- **人员：** 后端工程师 1-2 人、算法/NLP 工程师 1 人（RAG 调优）、前端工程师 0.5 人（Streamlit/Gradio 即可）

## The Assignment

在开始写代码之前，做三件事：

1. **影子观察（Shadowing）：** 花一个下午坐在一个产品经理和一个开发工程师旁边，看他们实际怎么查文档、搜代码、查数据。记录具体操作序列和耗时。这会在你设计 UI 和工作流时反复提醒你真实的使用模式，而不是你想象中的模式。

2. **数据源盘点：** 整理三种数据源的具体清单——PRD 文档存在哪（Confluence/飞书/其他）、代码仓库数量和语言（Python/Java/Go）、数据库类型和规模（PG/MySQL、表数量级）。这直接决定解析器选型和索引策略。

3. **搭建 Text-to-SQL 原型：** 用 PGVector + BGE-M3 + Claude API + Streamlit，先跑通一个只做 SQL 自然语言查询的最小原型。让一个真实的 PM 同事用它查一次数据。你看他们第一反应就知道这条路对不对。
