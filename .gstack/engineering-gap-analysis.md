## 总体评价

SPMA 设计文档在架构层面做得非常出色——Supervisor-Worker 模式、实体完备度评估、查询改写流水线、Code Worker 双引擎检索、SQL 正确性保障的四层保护——这些核心设计决策都经过了充分的推敲和反事实论证。2131 行的设计文档在"做什么"和"为什么这样做"上做到了罕见的深度。

但**"怎么做"的工程层面有显著缺口**。以下是按严重程度排列的缺失项。每个缺口标注影响面和优先级。

## 🔴 P0 — 不解决就无法上线

### 1. ~~没有 API 契约定义~~ ✅ 已补齐

**状态：** 已于 2026-06-07 补齐。详见 [api-contract.yaml](api-contract.yaml) + [api-design-decisions.md](api-design-decisions.md)。

**补齐内容：** 17 个 endpoint、30+ 错误码、8 个核心 Schema、SSE 流式协议、SQL 确认闸门、设计决策文档。

**缺什么：**
```
POST   /api/v1/query          — 核心查询接口，Request/Response schema 未定义
GET    /api/v1/query/{id}     — 查询历史
GET    /api/v1/health         — 健康检查
POST   /api/v1/feedback       — 用户反馈（设计提了但没接口）
GET    /api/v1/sources/status — 数据源新鲜度
```

**具体缺失：**
- Request schema：`query_text`、`session_id`、`max_sources`、`timeout_ms`、`stream` 字段的类型/必填/默认值
- Response schema：`answer`、`sources[]`、`citations[]`、`degradation_level`、`latency_ms`、`suggested_followups[]`
- 错误响应格式：统一的 `{"error": {"code": "...", "message": "...", "details": {...}}}` 结构
- 分页/流式响应的协议选择（SSE vs WebSocket vs chunked transfer）

**影响：** 前后端无法并行开发。没有 API 契约，测试无法写 mock，CI 无法做契约测试。

**建议：** Phase 1 启动前用 OpenAPI 3.1 定义至少 `/query`、`/health`、`/sources/status` 三个接口。Phase 2 补充 `/feedback`、查询历史、管理接口。

### 2. 没有错误码分类体系

**现状：** 设计文档 4 次提到"不静默失败"，但没有任何具体的错误码定义。错误处理分散在各处——SQL Guard 有校验失败、Worker 有超时、LLM 有 API 错误——但这些没有统一的命名和分类。

**缺什么：**

```
E0000-E0999  系统级错误
  E0001  InternalError           — 未分类的内部异常
  E0010  ServiceUnavailable      — 全部后端不可用（L4触发）
  E0020  PartialDegradation      — 部分Worker不可用
  E0030  RateLimited             — 触发限流

E1000-E1999  查询理解层错误
  E1001  ClassificationFailed    — 意图分类失败（LLM超时/不可用）
  E1002  EntityExtractionFailed  — 实体抽取失败
  E1010  AmbiguousQuery          — 无法确定意图，需要反问用户
  E1011  QueryTooShort           — 查询过短且无上下文

E2000-E2999  Worker层错误
  E2001  DocWorkerTimeout        — Doc Worker 超时
  E2002  CodeWorkerTimeout       — Code Worker 超时
  E2003  SQLWorkerTimeout        — SQL Worker 超时
  E2010  NoResultsFound          — 三源均无结果
  E2020  PartialResults          — 部分源有结果（非错误，信息性）

E3000-E3999  SQL Worker 错误
  E3001  SQLGenerationFailed     — LLM 生成 SQL 失败
  E3002  SQLGuardRejected        — SQL Guard 拦截（不安全SQL）
  E3003  SQLExecutionFailed      — 只读副本执行失败
  E3004  SQLSelfHealingExhausted — 自修复循环3次后仍失败

E4000-E4999  数据源错误
  E4001  DocSourceUnavailable    — PRD文档源不可达
  E4002  CodeRepoUnavailable     — 代码仓库不可达
  E4003  DatabaseUnavailable     — 数据库只读副本不可达
  E4010  StaleData               — 数据可能过期（索引未更新）

E5000-E5999  认证/授权错误
  E5001  Unauthenticated         — 未登录
  E5002  Forbidden               — 无权限
  E5003  SessionExpired          — 会话过期
```

**每个错误码必须附带：** HTTP status code、user-facing message（中文）、log message（英文结构化）、是否可重试、降级路径。

**影响：** 没有错误码，监控无法聚合告警、前端无法做差异化 UI（"重试"按钮 vs "联系管理员" vs "换种问法"）、on-call 无法快速定位。

### 3. 没有具体的监控仪表盘和告警规则

**现状：** 设计文档只提到"OpenTelemetry → Grafana + Langfuse"，没有任何 dashboard 定义或告警规则。

**缺什么——至少 4 个仪表盘：**

**Dashboard 1: 服务质量概览（墙上大屏）**
- QPS（实时 + 24h趋势）
- P50/P95/P99 延迟（按查询类型分面：单源/跨源、按 Worker 分面）
- 错误率（按错误码分面）
- 降级触发次数（L1/L2/L3/L4 分别计数）
- 可用性百分比（滚动30天）

**Dashboard 2: LLM 成本与质量**
- Token 用量（按模型分：Sonnet/Haiku/Qwen3）
- 成本（按天/按查询分面）
- 幻觉率趋势（Ragas 评估）
- Prompt Cache 命中率
- 分类准确率趋势

**Dashboard 3: 数据新鲜度**
- Doc/Code/SQL 索引延迟（分钟，目标 <5/<5/<10）
- 索引任务成功率
- 各源文档/代码/表数量变化趋势
- Webhook 投递成功率

**Dashboard 4: 用户体验**
- 查询满意度分布（👍/👎 + 无反馈）
- 反问率（系统反问用户的查询占比——希望这个数下降）
- bare-entity 查询占比（希望这个数下降）
- 用户修正率（同一 session 内用户改变 query 重查的比例）
- 活跃用户数趋势

**告警规则（至少 8 条）：**
```yaml
- alert: HighErrorRate
  expr: error_rate_5m > 0.05
  severity: critical
  
- alert: HighLatencyP99
  expr: latency_p99_single_source_5m > 3000 or latency_p99_cross_source_5m > 8000
  severity: warning

- alert: LLMPrimaryDown
  expr: llm_primary_error_rate_5m > 0.3
  severity: critical  # L1 降级应该已经自动触发，告警是通知人

- alert: DegradationTriggered
  expr: degradation_level > 0
  severity: warning  # 只要不是在L0就应该有人知道

- alert: IndexStaleness
  expr: doc_index_lag_minutes > 10 or code_index_lag_minutes > 10
  severity: warning

- alert: ZeroResultsRate
  expr: zero_results_rate_1h > 0.2
  severity: info

- alert: SQLGuardRejectionSpike
  expr: rate(sql_guard_rejected_total[5m]) > 10
  severity: warning

- alert: FeedbackNegativeSpike
  expr: rate(user_feedback_negative_total[30m]) > rate(user_feedback_positive_total[30m]) * 2
  severity: warning
```

**影响：** 没有这些，99.9% 可用性目标是一句空话——你甚至不知道你是否达标。

## 🟡 P1 — 上线前必须解决，但可以用简化版

### 4. 没有 LLM 成本估算和预算控制

**现状：** 提到用 Claude Sonnet 4 + Haiku + Qwen3-32B，但零成本分析。

**缺什么：**

| 调用环节 | 模型 | 每次预估 Token | 频率假设 | 日成本估算 |
|---------|------|--------------|---------|-----------|
| 意图分类+实体抽取 | Haiku | ~500 input + 100 output | 100次/天 | $0.015 |
| 查询改写-扩展 | Haiku | ~300 input + 50 output | 70次/天 | $0.007 |
| 查询改写-分解 | Haiku | ~400 input + 200 output | 30次/天 | $0.006 |
| 查询改写-HyDE | Haiku | ~200 input + 300 output | 20次/天 | $0.004 |
| Worker 检索结果摘要 | Sonnet | ~2000 input + 500 output | 100次/天 | $0.75 |
| Synthesis 最终回答 | Sonnet | ~3000 input + 800 output | 100次/天 | $1.20 |
| SQL 生成+自修复 | Sonnet | ~1500 input + 300 output | 60次/天 | $0.54 |

**估算：日均 ~$2.5，月均 ~$75（不含本地 Qwen3 的 GPU 成本）。** 这是 100 QPD（query per day）场景。如果 QPS 增长 10x，注意 Sonnet 调用是瓶颈。

**还需要定义：**
- 单次查询的 token 预算上限（防止恶意或异常的超级长查询）
- 单用户日 token 限额
- Prompt Caching 策略（哪些 prompt prefix 可以缓存？预计节省多少？）

**影响：** 不估算，上线后第一张 Claude API 账单可能是惊吓。而且 LLM 成本直接影响系统架构决策——如果 Sonnet 合成占成本的 70%，优化 prompt 或换 Haiku 做合成是合理的工程选择。

### 5. Prompt 版本管理缺失

**现状：** 设计文档中 prompts 作为代码字符串散落各处（python 代码块中的 `CLASSIFICATION_PROMPT`、`EXPANSION_PROMPT`、`DECOMPOSE_PROMPT`、`HYDE_PROMPT` 等），没有版本管理策略。

**缺什么：**

```
prompts/
├── classification/
│   ├── v1.yaml          # 当前版本
│   └── v1.1.yaml        # 改进版本
├── expansion/
│   └── v1.yaml
├── decomposition/
│   └── v1.yaml
├── hyde/
│   └── v1.yaml
├── sql_generation/
│   └── v1.yaml
├── synthesis/
│   └── v1.yaml
└── tests/
    ├── classification_test_cases.jsonl
    └── expected_outputs/
```

每个 prompt 文件包含：
```yaml
version: "1.0"
model: "claude-haiku-4.5"
temperature: 0.0
max_tokens: 500
system_prompt: |
  ...
user_template: |
  查询: {query}
evaluation:
  test_set: "classification_test_cases.jsonl"
  min_accuracy: 0.95
changelog:
  - version: "1.0"
    date: "2026-06-01"
    change: "初始版本，基于100条标注数据"
```

**还需要：**
- Prompt 变更的 CI 检查——改 prompt → 自动跑评估集 → 准确率下降 >2% 自动 block 合并
- Prompt 的 git 历史（YAML 文件天然支持）
- 线上 prompt 切换能力（热加载，不重启服务）

**影响：** 不改 prompt 版本管理，三个月后你会有一堆"不知道哪个版本在用"的 prompt 字符串散落在代码里，调优靠 grep，回滚靠记忆。

### 6. 没有完整的数据库 Schema 和迁移策略

**现状：** 只在设计文档中作为示例出现了一张 `code_chunks` 表的 DDL。`doc_chunks`、`schema_chunks`、`query_log`、`feedback`、`sessions`、`synonym_map`、`audit_log` 等表完全没有定义。

**缺什么（至少 7 张表）：**

```sql
-- 1. doc_chunks（PRD文档分块）
-- 2. code_chunks（已有DDL但不完整，缺少 embedding 列和索引）
-- 3. schema_chunks（SQL Schema分块 + 业务元数据）
-- 4. query_log（每次查询的完整记录）
-- 5. feedback（用户反馈）
-- 6. sessions（对话会话，多轮对话需要）
-- 7. synonym_map（同义词映射表，热配置）
-- 8. audit_log（审计日志，不可变）
```

**还需要：** Alembic 迁移脚本策略、Schema 变更的 CI 检查（迁移脚本是否可逆？是否与代码版本一致？）、蓝绿部署时如何处理 Schema 变更（先加列再删列，不破坏旧版本读取）。

**影响：** 没有完整的 Schema 设计就开始写代码，会在 Phase 2/3 频繁改表结构，每次都是 downtime 风险。

### 7. 没有用户反馈闭环设计

**现状：** 设计文档提到 NPS 和用户满意度，但反馈机制的设计是零散的——有些地方提到"用户反馈"但没有具体的 UI 交互、数据模型、反馈分类、改进流程。

**缺什么：**

**反馈入口（前端）：**
- 每条回答下方的 👍/👎 按钮（附带可选原因：不准确/不完整/无关/太慢）
- "这个答案对你有帮助吗？"提示（用户未主动点反馈时，30% 概率弹出——A/B 测试）

**反馈分类和消费：**
```
用户点 👎 → 记录 query + 系统回答 + 检索结果 + Worker 状态
          → 原因="不准确" → 人工 review 队列（或 LLM 预审）
          → 原因="太慢"   → 对照延迟监控，可能触发自动排查
          → 原因="不相关" → 意图分类或实体抽取可能有问题
```

**反馈到改进的闭环：**
- 每天 review 前一天的所有 👎
- 每周汇总 → 分类问题排名 → 驱动下一周的改进优先级
- 反馈信号直接输入 Ragas 评估流程（人工标注往往太少，用户反馈是免费的标注信号）

**影响：** 没有反馈闭环，RAG 系统会退化——你永远不知道用户看到了什么、满意不满意。50 条 ground truth 是冷启动，用户每天的点击才是持续的改进燃料。

## 🟢 P2 — Phase 2/3 必须解决

### 8. 没有灾难恢复（DR）计划

**现状：** 零备份策略、零 RPO/RTO 目标。

**缺什么：**
- PGVector 备份策略：`pg_dump` + WAL 归档？频率？异地存储？
- Redis 备份策略：RDB snapshot 频率？AOF？
- RPO（Recovery Point Objective）：最多丢失多少数据？建议 1 小时
- RTO（Recovery Time Objective）：恢复需要多长时间？建议 4 小时
- 恢复演练计划：每季度一次
- PGVector 重建方案：如果向量索引损坏，从源数据全量重建需要多长时间？

**影响：** 99.9% 可用性 = 每月最多 43 分钟不可用。数据丢失恢复时间不算在这个预算里——如果数据库挂了，恢复要 8 小时，当月可用性直接崩到 98.9%。

### 9. 没有前端/UI 设计

**现状：** 只提到"Streamlit/Gradio 即可"，但设计文档描述的交互复杂度远超 Streamlit 的承载能力。

**Streamlit 做不到的事情：**
- 多轮对话（设计文档有多轮上下文改写）
- 反问用户后的输入框（分类置信度低时需要反问）
- SQL 执行前的确认闸门 UI
- 引用标注的可点击跳转（`[PRD §3.2]`、`[src/auth/oauth.py:42]`）
- 数据新鲜度标注的视觉呈现
- 反馈按钮的 A/B 测试
- SSO 登录 + 8h JWT session 管理
- 审计日志查看（管理界面）
- 同义词映射审核界面（设计文档描述了详细的审核 UI）

**建议：** v1 可以用 Streamlit 做 MVP UI，但必须接受：所有设计文档中描述的交互细节在 Streamlit 上都是妥协版。v2 应该用 Next.js 或类似框架做正式前端。如果你们决定用 Streamlit 交付整个 v1，需要在 Phase 1 启动前明确哪些交互特性要砍掉。

### 10. 没有具体的运维 Runbook

**现状：** 零操作手册。

**缺什么（至少 5 份 Runbook）：**

1. **服务重启流程：** 各服务的依赖顺序（PGVector → Redis → Workers → Supervisor → API Gateway），重启后的健康检查命令
2. **数据重索引流程：** 如何触发全量重索引？预计耗时？（这直接决定 RTO）
3. **密钥轮换流程：** Claude API Key、数据库密码、OIDC Client Secret 的轮换步骤（不能中断服务）
4. **故障响应流程：** Severity 1/2/3/4 分级、升级路径、on-call 联系方式、事后复盘模板
5. **容量扩展流程：** 如何加 K8s Worker 副本？如何扩展 PGVector 读副本？触发条件（CPU > 70%? P99 > 3s?）

**影响：** 凌晨 3 点系统挂了，on-call 工程师面对 6 份设计文档，不知道哪个服务先重启。

## 🔵 P3 — 上线后持续完善

### 11. 没有安全威胁模型

设计文档有安全考虑的散点（脱敏、只读副本、SSO、审计日志），但没有结构化的威胁建模。

**建议：** Phase 2 前完成一次 STRIDE 威胁建模，聚焦：
- Prompt Injection（用户输入包含"忽略之前指令"）
- 间接 Prompt Injection（PRD 文档中包含恶意指令，被检索到后注入 LLM 上下文）
- 数据泄露（脱敏层漏过、审计日志泄露敏感内容）
- 拒绝服务（恶意高频查询、超大时间范围 SQL）

### 12. 没有供应商锁定风险评估

系统深度依赖：Claude API（Anthropic）、LangGraph（LangChain）、PGVector、BGE-M3（BAAI）。

**建议：** 识别"最难替换"的依赖（LangGraph 可能排第一——Supervisor-Worker 的 checkpoint 和状态管理深度绑定），评估每个依赖的替换成本和迁移方案。不是现在做——但要写在风险登记册里。

### 13. 没有多环境管理策略

**缺什么：** dev/staging/prod 环境的差异定义——数据量差异？LLM 用的是真实 API 还是 mock？数据库只读副本指向哪？dev 环境能不能访问生产数据（脱敏后）？

### 14. 没有数据保留和清理策略

**缺什么：** 嵌入向量的生命周期（旧代码删了，它的 embedding 什么时候删？）、查询日志保留多久、审计日志保留多久（合规需求）、Redis 缓存的淘汰策略（LRU？TTL 目前定义了但不完整）。

### 15. 没有负载测试计划

**现状：** 性能目标定义得很好（P99 < 3s/5s/8s），但没有任何关于如何验证的说明。

**缺什么：** Locust/k6 测试脚本、并发用户数假设（100个用户中同时查询的有几个？峰值 QPS？）、测试环境需要的数据规模（多少文档/代码/表）

以下内容设计文档里**有**，但藏在段落实例或代码注释中，没有以"这是工程交付物"的清晰形式呈现。提取出来即可：

| 隐藏内容 | 所在位置 | 应提取为 |
|---------|---------|---------|
| 实体权重体系（10/5/4/2/1） | 01-Supervisor 完备度评估 | `config/entity_weights.yaml` |
| 同义词映射表字段定义 | 01-Supervisor 查询改写 | `config/synonym_map_schema.yaml` |

| 高风险 SQL 确认闸门规则 | 04-SQL Worker | `config/sql_confirmation_rules.yaml` |

## 总结：按 Phase 补齐优先级

| Phase | 必须补齐 | 建议补齐 |
|-------|---------|---------|
