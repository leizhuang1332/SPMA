## Problem Statement

产品经理和开发工程师在企业内部面临严重的信息碎片化问题。三种核心数据来源——PRD 产品文档、开发代码、SQL 数据库——分散在不同的系统和工具中（Confluence/Wiki、IDE、数据库客户端），每次查询需要切换系统、手动搜索、人脑关联。这种"三系统切换"的模式每天浪费大量时间，导致：

- **产品经理**：查不到历史需求的完整上下文，做决策缺乏数据支撑
- **开发工程师**：故障定位时无法快速追溯代码到需求，查数据验证问题需要找 DBA 或手写 SQL
- **两拨人的共同痛点**：无法用自然语言直接查询业务数据

## Demand Evidence

- 具体的团队同事明确投诉过这个问题——有具体的人和场景，不是泛泛的"可能有用"
- 现状是多系统切换 + 人工搜索，每天切换 5-10+ 次
- 两拨用户群体都有明确的使用场景：PM 的需求溯源场景、开发的故障定位场景、共同的数据查询场景

## Status Quo

当前的"竞品"是碎片化的工具组合：

| 用户 | 现状 | 代价 |
|------|------|------|
| 产品经理 | Confluence/Wiki 搜索 PRD 文档 | 找不到历史版本、无法对比、需要人脑关联需求变更 |
| 开发工程师 | IDE grep 搜索代码 + 手动追踪需求 | 故障定位慢、新人上手周期长（依赖老员工口传） |
| 双方共同 | 找 DBA 或手写 SQL 查询数据 | 瓶颈在 DBA 带宽、非技术人员完全无法自助 |

信息获取依赖人际网络——新人入职要几个月才能独立工作。这是企业知识管理的典型失效模式。

## Target User & Narrowest Wedge

**核心用户画像：**
- 产品经理：需要需求溯源能力——某个功能为什么这么做？历史版本怎么演化的？这个改动影响了哪些下游？
- 开发工程师：需要快速定位能力——这个 bug 跟哪个需求相关？涉及哪些代码模块？影响了哪些数据表？

**Wedge：** 三源全覆盖 + 企业级工程化——用户明确要求不是 MVP 原型，而是从第一天就按生产标准设计的完整系统。对 intrapreneurship 场景来说，这是合理的：内部工具如果不可靠，用户不会用第二次。

## Constraints

- 企业内网部署，敏感数据不出企业边界。LLM 调用必须经过数据脱敏层（脱敏后可通过外网 API），或使用本地部署模型
- 三种数据源差异极大（自然语言文档、结构化代码、关系型数据库），不能用一套检索策略覆盖
- 内部用户对响应延迟敏感：P99 < 3 秒（单源查询）、P99 < 5 秒（跨源查询）
- 需要支持中文内容（PRD 文档、自然语言查询、中文代码注释）
- 系统可靠性决定用户信任——一次查询失败可能导致用户放弃使用

## Premises

1. **三种数据源需要三种不同的检索策略** — PRD 文档用语义搜索+关键词混合检索，代码用 AST 感知的结构化搜索，SQL 用 Schema 感知的 RAG 检索。不能用一套策略打天下。

2. **Supervisor-Worker 智能体架构** — 一个路由层理解用户意图，分发到专门的 Worker（文档 Worker / 代码 Worker / SQL Worker），各自独立检索，最后由合成层汇总。跨源查询是核心价值——不是锦上添花。

3. **四级降级保障** — 全功能 → LLM 降级（切换备用模型）→ 向量检索降级（纯关键词）→ 缓存兜底（热点问答）→ 静态 FAQ 兜底。每一级都是自动触发、自动恢复。

4. **Text-to-SQL 必须包含 SQL Guard 层** — 语法校验 + DDL/DML 拦截 + 只读副本执行 + 自修复循环。生产环境中不允许任何写操作风险。

5. **三源平等支柱** — PRD 文档、代码、SQL 是三个同等重要的检索入口。用户坚持了这一点（在第二意见挑战后维护了原始立场），因为跨源溯源（"这个需求影响了哪些代码和哪些表？"）是最核心的使用场景。

## Cross-Model Perspective

独立 AI 审查（Claude 子 Agent）提供了以下视角：

**Steelman（最强版本）：** "企业溯源引擎"——不是三个独立的搜索框，而是一个理解"需求如何变成代码、代码如何触及数据"的统一查询层。PM 问"这个 PRD 变更影响了哪些表？"，开发问"这个 bug 关联哪个需求的哪个版本？"——这些跨源问题才是 RAG 独有的价值。

**被挑战的前提：** 第二意见认为 Supervisor-Worker 对第一版过度复杂，建议先用分类器+顺序管道，等日志显示 >40% 查询需要跨源时再升级。用户的回应：坚持原始前提——因为跨源查询就是核心场景，从一开始就需要 Supervisor 编排。

**48 小时原型建议：** Text-to-SQL + PRD 上下文（跳过代码搜索），用 PGVector + BGE-M3 + Streamlit。正则表达式 SQL Guard 足够。零降级。

**综合：** 第二意见在"从 SQL 开始"这个方向上是正确的——SQL 是最大公约数。但它对 Supervisor 模式的反对在用户的场景下不成立，因为跨源查询不是边缘情况而是核心价值主张。

## Approaches Considered

### Approach A: Supervisor-Worker Agentic RAG ✅ 选中
**Summary:** 智能路由层理解意图 → 分发到三个专门化 Worker（文档/代码/SQL 各自用最优检索策略）→ 跨源合成层汇总。支持多跳推理、4 级降级、完整可观测性。
**Effort:** 人：5-7.5 人·月（2-3 人并行约 3-5 自然月）；AI 辅助可加速代码生成但无法缩短架构决策和数据源对接时间
**Risk:** 中（Agent 编排的复杂性需要精心设计 fallback）
**Pros:**
- 每种数据源用最优检索策略，不妥协
- 天然支持跨源多跳查询（核心价值场景）
- 易于扩展第四种、第五种数据源
- 业界验证的成熟模式（Protocol-H, LangGraph Supervisor）
**Cons:**
- Agentic 编排增加延迟和工程复杂度
- 需要精心设计回退路径和超时处理
- 调试和可观测性要求高于简单管道

### Approach B: 联邦并行检索 RAG
**Summary:** 不做智能路由——每个查询同时发到三个专门化检索管道，各自独立返回，用 RRF（Reciprocal Rank Fusion）融合排序后统一展示。
**Effort:** 人：~2-3 个月 / AI 辅助：~1-2 周
**Risk:** 低
**Pros:**
- 延迟更低（并行执行，无路由决策开销）
- 架构简单，调试容易
- 每条管道独立优化、独立故障隔离
**Cons:**
- 不支持多跳跨源推理（"先查文档再决定查哪个表"）
- 简单查询也要跑三条管道，资源浪费
- 无法动态调整检索策略

### Approach C: 知识图谱 + RAG 混合
**Summary:** 构建统一知识图谱——PRD 需求节点、代码模块节点、数据库表节点用关系边连接。RAG 做初始检索，图遍历做关系追踪。
**Effort:** 人：~4-6 个月 / AI 辅助：~3-4 周
**Risk:** 高
**Pros:**
- 最强的跨源溯源能力（确定性关系查询，不会幻觉）
- 对"影响分析"类问题最优
- 图结构天然支持版本演化和时间线查询
**Cons:**
- 构建和维护知识图谱成本极高
- 需要持续同步三种数据源的变化（CDC 链路）
- 冷启动问题（初始图谱构建工作量大）

## Recommended Approach

**Approach A: Supervisor-Worker Agentic RAG**

### 意图分类器设计

这是整个 Supervisor 的入口——分类错了，后面的 Worker 调度全错。不需要训练专用模型，用 **LLM 结构化分类 + 规则兜底** 两层架构。

#### 第一层：LLM 结构化分类

使用轻量模型（Claude Haiku 或本地 Qwen3-8B，延迟 < 500ms），通过 LangChain `with_structured_output` / JSON mode 输出固定 schema：

```python
from pydantic import BaseModel
from typing import Literal, Optional

class ClassificationResult(BaseModel):
    """Supervisor 意图分类输出"""
    sources: list[Literal["doc", "code", "sql"]]
    is_cross_source: bool
    entities: Optional[dict]  # {"module": "用户登录", "req_ids": ["REQ-187"], ...}
    query_type: Literal["trace", "search", "data_query", "explain"]
```

分类 Prompt：

```
你是一个查询路由器。分析用户问题，输出 JSON。

数据源定义：
- doc: PRD 文档、产品需求、功能规格、需求变更
- code: 代码实现、函数、类、文件路径、bug、架构
- sql: 业务数据、统计、报表、指标查询

分类规则：
- 含需求ID [REQ-XXXXX] → 至少包含 doc
- 含表名/列名/数据/统计/多少 → 至少包含 sql
- 含文件路径/函数名/代码/类名/bug → 至少包含 code
- "X影响Y"、"X对应哪个Z" → 跨源查询，标记 is_cross_source=true
- 模糊查询无法判断 → 默认三源全查，标记 query_type="search"

示例：
"上周用户登录改了什么需求？" → {"sources": ["doc"], "is_cross_source": false, ...}
"REQ-187 改了哪些代码和表？" → {"sources": ["doc","code","sql"], "is_cross_source": true, ...}
"过去7天新增了多少用户？" → {"sources": ["sql"], "is_cross_source": false, ...}
"oauth.py 的 token_refresh 是谁写的？" → {"sources": ["code"], "is_cross_source": false, ...}
"用户登录怎么做的？" → {"sources": ["doc","code","sql"], "is_cross_source": true, "query_type": "search", ...}
```

#### 第二层：规则兜底

LLM 分类结果之后加硬规则——LLM 做语义理解，规则做确定性补刀：

```python
import re

def validate_and_correct(result: ClassificationResult, user_input: str) -> ClassificationResult:
    """规则层：修正 LLM 分类的确定性问题"""
    
    # 规则1: 提到了表/字段/数据查询语义 → 强制加 sql
    if re.search(r'(表|字段|列|数据|统计|多少|汇总|趋势|报表|指标)', user_input):
        if "sql" not in result.sources:
            result.sources.append("sql")
            result.is_cross_source = len(result.sources) > 1
    
    # 规则2: 包含需求ID格式 [REQ-\d+] 或 PROJ-\d+ → 强制加 doc
    if re.search(r'(REQ|PROJ|PRD)-\d+', user_input):
        if "doc" not in result.sources:
            result.sources.append("doc")
            result.is_cross_source = len(result.sources) > 1
    
    # 规则3: 包含文件路径或函数/类引用 → 强制加 code
    if re.search(r'\.(py|java|go|ts|js)\b|def |func |class |函数|方法|类\b', user_input):
        if "code" not in result.sources:
            result.sources.append("code")
            result.is_cross_source = len(result.sources) > 1
    
    # 规则4: 极短查询（<5个字）且无任何实体 → 保守策略：三源全查
    if len(user_input) < 5 and not result.entities:
        result.sources = ["doc", "code", "sql"]
        result.is_cross_source = True
        result.query_type = "search"
    
    return result
```

#### 分类质量保障

**评估集：** 从 Shadowing 观察和用户测试中收集 100 条真实查询 → 人工标注正确的 `sources` → 跑分类器计算准确率。目标：≥ 95%。

分类准确率在这个场景下不是高难度问题——三类数据源的语义边界足够清晰：

| 失败模式 | 例子 | 发生率估计 | 对策 |
|---------|------|-----------|------|
| 模糊查询 | "用户登录怎么做的？" | ~10% | 默认三源全查，答案多样性本身就是价值 |
| 术语歧义 | "表的索引" | ~2% | 规则优先——含"表"字就走 SQL Worker |
| 无上下文短查询 | "那个功能" | ~5% | 多轮对话带上前一轮分类结果；单轮直接反问 |
| 跨源漏判 | "REQ-187 的实现"（漏了 code） | ~3% | 规则2补上；同时一旦识别需求ID，默认同时查 doc+code |

**持续改进：** 每次分类结果记录日志（用户问题 + 分类输出 + 实际路由 + 用户反馈）。每周 review 20 条，往规则层加 pattern。不依赖一次性调参。

#### 降级路径

- **Haiku API 不可用** → 切到本地 Qwen3-8B（分类质量下降 2-3%，规则层兜底）
- **全部 LLM 不可用** → 纯规则分类（正则匹配 + 关键词），准确率约 85%，但系统仍可用
- **分类置信度极低**（无实体、无关键词）→ 保守策略：三源全查

### 实体抽取设计

意图分类回答"查哪些源"，实体抽取回答"在每个源里查什么"。分类和抽取共享同一个 LLM 调用——一次请求同时输出分类结果和实体列表，避免串行延迟。

#### 抽取哪些实体

```python
class ExtractedEntities(BaseModel):
    """从用户问题中抽取的结构化实体"""
    
    # ── 跨源通用实体 ──
    module: Optional[str]          # 功能模块："用户登录"、"支付网关"、"审批流"
    req_ids: list[str]             # 需求ID：["REQ-2024-0187", "PROJ-1234"]
    time_range: Optional[str]      # 时间范围："上周"、"过去7天"、"2026年3月"
    version: Optional[str]         # 版本/分支："v2.3"、"release/2026Q1"、"main"
    
    # ── SQL 相关实体 ──
    table_names: list[str]         # 表名：["users", "orders"]
    column_names: list[str]        # 列名：["status", "amount", "created_at"]
    metrics: list[str]             # 指标/聚合："新增用户数"、"订单总额"、"日活"
    group_by: Optional[str]        # 分组维度："按天"、"按部门"、"按状态"
    
    # ── 代码相关实体 ──
    code_refs: list[str]           # 文件/类/函数引用：["oauth.py", "TokenService", "login_oauth"]
    person: Optional[str]          # 代码作者/提交人："张三"、"@leizhuang1332"
    
    # ── 文档相关实体 ──
    doc_types: list[str]           # 文档类型：["PRD", "技术方案", "接口文档", "会议纪要"]
```

**抽取 Prompt（与意图分类同一请求）：**

```
从用户问题中抽取以下实体。找不到的字段设为 null 或空列表。

实体说明：
- module: 问题涉及的功能模块或业务领域
- req_ids: 需求编号（格式 REQ-XXXXX 或 PROJ-XXXXX）
- time_range: 时间范围表述。保留原文，如"上周"、"过去7天"、"2026年Q1"
- version: 版本号或分支名
- table_names: 提到的数据库表名（英文或中文表名）
- column_names: 提到的列名/字段名
- metrics: 要查询的指标，如"新增用户数"、"订单总额"
- group_by: 分组维度，"按天"→daily，"按部门"→department
- code_refs: 提到的文件名、类名、函数名
- person: 提到的人名（代码作者/提交人）
- doc_types: 提到的文档类型，如"PRD"、"技术方案"、"接口文档"

示例：
"过去7天订单表里各种状态的订单数是多少？"
→ {"module":"订单","time_range":"过去7天","table_names":["orders"],"column_names":["status"],"metrics":["订单数"],"group_by":"按状态"}

"张三上周改的 oauth.py 里 TokenService 这个类，关联的需求是哪个？"
→ {"req_ids":[],"code_refs":["oauth.py","TokenService"],"person":"张三","time_range":"上周","module":"认证"}
```

#### 抽取后怎么用

实体是下发给各 Worker 的检索参数。不同的实体到达不同的 Worker，各自用最擅长的方式消费：

```
                        ┌─────────────────────────────┐
                        │     ExtractedEntities        │
                        └─────────────┬───────────────┘
                                      │
            ┌─────────────────────────┼─────────────────────────┐
            │                         │                         │
            ▼                         ▼                         ▼
    ┌───────────────┐       ┌───────────────┐       ┌───────────────┐
    │   Doc Worker   │       │  Code Worker   │       │   SQL Worker   │
    └───────────────┘       └───────────────┘       └───────────────┘
```

**Doc Worker：**

| 实体 | 用法 | 示例 |
|------|------|------|
| `req_ids` | 精确匹配——元数据过滤 `WHERE req_id IN (...)`，这是 Doc Worker 最高优先级的检索信号 | `req_id = "REQ-2024-0187"` → 直接返回该需求的所有 PRD 片段 |
| `module` | 语义搜索+关键词——向量检索的 query text | `"用户登录 需求规格 PRD"` |
| `time_range` | 元数据过滤——限定文档更新时间范围 | `WHERE updated_at > '2026-05-29'` |
| `version` | 版本过滤——Confluence 版本历史或 Git tag 对应的文档快照 | `WHERE version = 'v2.3'` |
| `doc_types` | 文档类型过滤 | `WHERE doc_type IN ('PRD', '技术方案')` |

**Code Worker：**

| 实体 | 用法 | 示例 |
|------|------|------|
| `code_refs` | 精确搜索——文件名精确匹配 + 函数名/类名 grep。最高优先级信号 | 搜 `oauth.py` 文件 + grep `TokenService` 定义 |
| `req_ids` | 关联搜索——grep commit log 和代码注释中引用该需求 ID 的代码 | `git log --grep="REQ-2024-0187"` 找变更文件 |
| `module` | 语义搜索——代码嵌入向量检索相关函数/类 | BGE-M3 检索 "authentication login oauth" |
| `person` | 作者过滤——`git log --author="张三"` 限定变更范围 | 结合 `time_range` 精确定位 |
| `time_range` | 时间过滤——限制 git log 的 `--since` / `--until` | `git log --since="2026-05-29"` |
| `version` | 分支/tag 过滤 | `git log release/2026Q1` |

**SQL Worker：**

| 实体 | 用法 | 示例 |
|------|------|------|
| `table_names` | 精确 Schema 检索——跳过语义搜索，直接查这些表的 DDL 和列注释 | `SELECT column_name, data_type FROM information_schema.columns WHERE table_name IN ('users','orders')` |
| `column_names` | 列级检索——在 Schema RAG 中精确定位列的语义描述 | 搜 `status` 列的业务含义（可能有枚举值映射："待支付/已支付/已取消"） |
| `metrics` | 聚合函数构造——决定 SELECT 子句 | `"订单数"` → `COUNT(order_id)`, `"订单总额"` → `SUM(amount)` |
| `time_range` | WHERE 条件——自然语言时间转 SQL 日期范围 | `"过去7天"` → `WHERE created_at >= NOW() - INTERVAL '7 days'` |
| `group_by` | GROUP BY 构造——自然语言分组转 SQL | `"按状态"` → `GROUP BY status`；`"按天"` → `GROUP BY DATE(created_at)` |
| `module` | Schema RAG 语义搜索——当 `table_names` 为空时，用 module 语义检索相关表 | `"用户登录"` → 检索 `users`, `user_sessions`, `oauth_states` 表 |

#### 跨源关联的桥梁实体

三类数据源之间靠两个实体互相"穿针引线"：

**`req_ids` — 最强跨源桥梁：**

```
用户问题: "REQ-187 改了哪些代码和表？"
         │
         ├─ Doc Worker: req_id="REQ-187" → 返回 PRD 变更内容
         │
         ├─ Code Worker: req_id="REQ-187" → git log --grep → 找到变更的代码文件
         │                                 → 提取代码中引用的表名 → 反馈给 SQL Worker
         │
         └─ SQL Worker: req_id="REQ-187" → 在表注释/数据字典中搜索
                       + 从 Code Worker 获得表名 → 精确查表结构
```

**`table_names` — SQL ↔ Doc 桥梁：**

```
用户问题: "users 表加了 oauth_provider 字段是哪个需求要求的？"
         │
         ├─ SQL Worker: table_name="users", column_name="oauth_provider"
         │             → 查 DDL 变更历史、列注释
         │
         └─ Doc Worker: module + 从 SQL Worker 获得的变更时间
                       → 搜索对应时间段的 PRD 文档 → 定位需求
```

#### 抽取质量保障

- **评估集：** 与意图分类共享 100 条标注数据，额外标注实体 correctness（抽取的实体是否正确）和 completeness（是否遗漏关键实体）
- **目标：** 实体 correctness ≥ 90%，completeness ≥ 85%
- **关键失败模式：** 用户用同义词指代表名（"用户表" vs `users`），用简称指代模块（"登录" vs "用户登录认证"）。对策：在规则层维护一个同义词映射表（"用户表"→`users`、"订单表"→`orders`），不依赖 LLM 猜测

#### 关键实体缺失的处理策略

实体抽取不是每次都成功的。用户说"那个功能上线了吗"，既没有需求 ID 也没有表名——但系统不能就此放弃。三层递进式补救：

```
抽取结果 → 实体完备度评估 → 分层处置
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
        实体充足         部分缺失          几乎为空
        (>2个关键)     (有module无细节)   (模糊指代/闲聊)
```

##### 第一层：实体完备度评估（确定性代码，非 LLM）

**谁打分：** 一段纯 Python 函数，不是 LLM。不调用任何模型，不引入不确定性。LLM 已经做完了它那部分（实体抽取），完备度评估只做一件事——**数一下抽到了几个高价值实体**。让 LLM 来评判自己的输出质量是循环依赖，逻辑上不成立。

**打分标准——单一原则：信息增益（Information Gain）**

一个实体的权重 = 它能把搜索空间缩小多少倍。不靠直觉，靠对检索行为的实际影响：

| 权重 | 实体 | 搜索空间缩减 | 原理 |
|------|------|-------------|------|
| **10** | `req_ids` | ~10,000× | 一个需求 ID 直接定位到 3-5 个文档片段，从百万级分块中精准捞出目标 |
| **10** | `table_names` | ~10,000× | 跳过整个 Schema RAG 语义搜索环节，`WHERE table_name='users'` 直接返回 DDL |
| **10** | `code_refs` | ~10,000× | `grep "TokenService"` + 文件名 `oauth.py`，毫秒级精确定位，不走向量检索 |
| **5** | `module` | ~100× | 把搜索从全量语料库缩小到一个功能域（如从 100 万块缩小到 1 万块） |
| **4** | `metrics` | ~100× | 确定了聚合方向，SQL Worker 不必猜测 SELECT 子句 |
| **4** | `column_names` | ~100× | 在 Schema RAG 中精确定位列，而非泛泛搜索整个表的列 |
| **2** | `time_range` | ~10× | 元数据过滤，缩小时间窗口内的候选集 |
| **2** | `person` | ~10× | `git log --author` 限定变更范围 |
| **2** | `version` | ~10× | 限定分支/tag，过滤无关版本的代码和文档 |
| **1** | `group_by` | ~2× | 确定分组维度，但对检索本身帮助有限 |
| **1** | `doc_types` | ~2× | 过滤文档类型，但大多数查询不指定类型 |

**为什么权重不是连续的？** 10、5、2 三档不是拍脑袋的——对应三种截然不同的检索行为：

- **权重 10（精确匹配级）：** 有了它，对应的 Worker 可以**跳过语义搜索**，直接用元数据查询或 grep。延迟从秒级降到毫秒级。
- **权重 5（语义锚点级）：** 不能跳过语义搜索，但把搜索限定在一个功能域内，检索精度大幅提升。
- **权重 1-2（辅助过滤级）：** 只在元数据层面做减法，对检索策略本身没有影响。语义搜索仍然需要跑。

**阈值逻辑：**

```python
def assess_completeness(entities: ExtractedEntities) -> str:
    """
    纯确定性代码。不调用 LLM，不做语义判断。
    
    权重设计原则：权重 = 搜索空间缩减倍数的对数。
    10 = 精确匹配级（可跳过语义搜索）
    5  = 强语义锚点级（大幅缩小搜索域）
    2  = 辅助过滤级（元数据减法）
    """
    ENTITY_WEIGHTS = {
        "req_ids": 10, "table_names": 10, "code_refs": 10,
        "module": 5,
        "metrics": 4, "column_names": 4,
        "time_range": 2, "person": 2, "version": 2,
        "group_by": 1, "doc_types": 1,
    }
    
    score = 0
    for field, weight in ENTITY_WEIGHTS.items():
        value = getattr(entities, field, None)
        if value and value != [] and value != "":
            score += weight
    
    # ≥10: 至少有一个精确匹配级实体 → 该Worker可以跳过语义搜索
    #  5-9: 有功能锚点但无精确匹配 → 混合检索（元数据过滤 + 语义搜索）
    #  <5: 无任何有效锚点 → 纯语义搜索兜底 或 反问用户
    if score >= 10:   return "rich"
    if score >= 5:    return "partial"
    return "bare"
```

**阈值为什么设在这里：**

- **rich ≥ 10：** 恰好等于一个精确匹配级实体的权重。意思是"至少有一个 `req_ids`、`table_names` 或 `code_refs` 被抽到了"。这三个任何一个命中，对应 Worker 的检索就从"大海捞针"变成"直接拿"。
- **partial ≥ 5：** 恰好等于一个 `module` 的权重。意思是"即使没有精确 ID，至少知道功能域"。此时检索有方向但不精确。
- **bare < 5：** 连功能域都不知道。检索没有锚点，纯语义搜索是唯一选择。

**为什么不用 LLM 打分：** LLM 评判自己的输出质量是一个经典的循环依赖问题——LLM 抽取了实体，然后问同一个 LLM"你觉得你抽得怎么样？"，它大概率会说"抽得很好"。反过来，如果换一个 LLM 来打分，又引入了新的延迟、成本和不确定性。而完备度评估本身不涉及语义判断——"有没有抽到需求 ID"是纯规则问题，用代码就够了。

##### 第二层：分级处置

**Level: rich（实体充足，约占 40% 查询）**

实体驱动精确检索。`req_ids` 非空→Doc Worker 直接用元数据过滤而非向量检索。`table_names` 非空→SQL Worker 跳过 Schema RAG 直接查 DDL。这条路延迟最低、准确率最高。

```
例: "REQ-187 改了 oauth.py 和 users 表的哪些字段？"
→ req_ids=["REQ-187"], code_refs=["oauth.py"], table_names=["users"], column_names=[]
→ Doc: WHERE req_id='REQ-187' （精确）
→ Code: grep "REQ-187" + 文件名匹配 oauth.py （精确）
→ SQL: WHERE table_name='users' （精确）
→ column_names 为空 → 返回 users 表全部字段供用户确认
```

**Level: partial（有方向缺细节，约占 45% 查询）**

实体提供检索锚点但不够精确。各 Worker 用实体做**混合检索**——实体用于元数据过滤缩小范围，原始用户问题用于语义搜索补全。

```
例: "用户登录模块上周的改动"
→ module="用户登录", time_range="上周", req_ids=[], code_refs=[], table_names=[]
→ Doc: 元数据过滤 time_range + 语义搜索 "用户登录 需求变更"
→ Code: 元数据过滤 time_range + 语义搜索 "authentication login" + git log --since
→ SQL: 无表名 → Schema RAG 语义搜索 "用户 登录 认证" 找相关表
→ 返回时标注"以下结果基于'用户登录'+'上周'检索，如需精确匹配请提供需求ID或表名"
```

**Level: bare（几乎为空，约占 15% 查询）**

没有实体锚点时，两种选择：

| 判断条件 | 动作 | 原因 |
|---------|------|------|
| 原始问题 > 15 字 | 三源全查，纯语义搜索，用原始问题直接做向量检索 | 有足够语义信息，检索质量不会太差 |
| 原始问题 ≤ 15 字 且 有对话历史 | 从上一轮分类和实体中继承上下文，合并后再检索 | "那个功能"→ 上一轮聊的是"用户登录"→ 用"用户登录"补全 |
| 原始问题 ≤ 15 字 且 无对话历史 | **反问用户**，不猜 | "你提到'那个功能'，请问是指哪个功能模块？方便给个需求ID或文件名吗？" |

```python
def handle_bare_entities(user_input: str, history: list, classification: ClassificationResult):
    """实体几乎为空时的处理"""
    
    if len(user_input) > 15:
        # 路径A: 原文够长，纯语义搜索兜底
        return {
            "strategy": "semantic_only",
            "query": user_input,  # 原始问题直接当检索query
            "sources": classification.sources,
            "note": "未识别到关键实体，使用全文语义搜索"
        }
    
    if history and history[-1].entities and history[-1].entities.module:
        # 路径B: 从对话历史继承module
        prev_module = history[-1].entities.module
        return {
            "strategy": "context_inherit",
            "query": f"{prev_module} {user_input}",  # "用户登录 那个功能怎么样了"
            "sources": classification.sources,
            "inherited_from": "previous_turn",
            "note": f"继承上轮上下文: module='{prev_module}'"
        }
    
    # 路径C: 无历史、无实体、短查询 → 反问
    return {
        "strategy": "ask_clarification",
        "prompt": "我理解你想了解更多细节，但需要你帮我确认一下——你指的是哪个功能模块？如果知道需求ID（如 REQ-XXXXX）或相关的文件名，告诉我会更精准。"
    }
```

##### 第三层：纯规则兜底（LLM 不可用时）

当 LLM 分类和抽取全部不可用时，降级到纯规则抽取——不需要模型，正则直接上：

```python
def rule_based_extraction(user_input: str) -> ExtractedEntities:
    """零 LLM 依赖的实体抽取，准确率约 70-80%"""
    return ExtractedEntities(

        module=extract_module(user_input),             # 从模块名词典匹配："登录"、"支付"、"订单"
```

#### 整体容错链路（端到端）

```
用户输入
    │
    ▼
LLM 意图分类 + 实体抽取 (Haiku, ~500ms)
    │
    ├─ 成功 ──→ 实体完备度评估 ──→ rich/partial/bare 分级处置
    │                                        │
    └─ 失败/超时 ──→ 纯规则分类+抽取 (~5ms)   │
                                              │
                                    ┌─────────┴─────────┐
                                    │   Worker 调度      │
                                    │   (使用实体 or     │
                                    │    原始query兜底)   │
                                    └────────────────────┘
```

**不静默失败的原则：** 当系统用了降级策略（语义搜索兜底、纯规则抽取、上下文继承）时，必须在返回给用户的回答末尾附加一个透明标注：

> ℹ️ 本次查询未识别到具体需求ID或表名，结果基于语义搜索，可能不够精确。[怎样提高搜索精度？](link-to-tips)

用户点击链接可以看到提示：提供需求 ID、表名、文件名或功能模块名可以获得更精准的结果。这既坦诚，也教育用户如何更好地使用系统。

#### 实体覆盖面：已知盲区和持续迭代

12 种实体类型不是从天而降的——它们是从三类数据源的检索需求中推导出来的。但这不意味着它们覆盖了所有查询。Schema-based 抽取的天然局限是：**你只能抽到预定义过的类型**。用户一定会问出超出 schema 的问题。

##### 已知不覆盖的查询类型

| 查询类型 | 示例 | 为什么当前实体不覆盖 | 当前兜底策略 |
|---------|------|--------------------|------------|
| **归因/根因分析** | "为什么用户登录突然变慢了？" | 12 种实体里没有"异常现象"或"根因"的载体 | module="用户登录" + 纯语义搜索"登录 变慢 原因"；三源全查 |
| **对比分析** | "v2.3 和 v2.4 的支付流程有什么差异？" | `version` 只存单值，无法表达"两个版本的差异" | version 字段存两个值会被 LLM 误判为单个；当前只能取一个版本，另一个靠用户追问 |
| **假设性/预演分析** | "如果要加 OAuth 登录，需要改哪些地方？" | 实体设计围绕"已存在的东西"，不涉及"计划中的变更" | module="用户登录" + code_refs=["oauth"]；Code Worker 是唯一的有效源 |
| **状态/进度查询** | "REQ-187 的开发进度到哪了？" | 实体不表达任务状态（Jira 状态、CI 状态等），这些数据不在三个源里 | req_ids=["REQ-187"] → Doc Worker 能找到需求内容，但无法回答"进度" |
| **依赖/影响链分析** | "改了 users 表的 email 字段会影响哪些服务？" | 实体抽取能拿到 `table_names` 和 `column_names`，但缺少"下游依赖"的载体 | SQL Worker 返回表结构 + Code Worker 搜索 `users.email` 引用 → 跨源关联有一定效果 |
| **操作指引** | "怎么给订单表建一个索引？" | 既不是查文档也不是查代码，是操作性知识 | 三源全查——Doc Worker 可能搜到 DBA 规范文档，Code Worker 搜到 migration 示例 |
| **概念解释** | "什么是 OAuth2.0 authorization code flow？" | 这甚至不一定是三种数据源里有的内容——可能是外部知识 | 三源全查；如果 PRD 里有相关背景说明就能回答，否则需要承认知识库不覆盖 |
| **多模块耦合分析** | "订单模块和支付模块的耦合点在哪？" | 单个 `module` 字段只能承载一个值 | module 取"订单+支付"；各 Worker 独立搜两个模块的交叉引用 |

##### 怎么评估覆盖面

不靠猜测，靠线上数据。评估分两个阶段：

**上线前（Shadowing 阶段）：**
```
1. 从 Shadowing 中收集 50-100 条真实用户问题
2. 逐条标注：
   - 这条查询能否被 12 种实体类型充分描述？
   - 如果能，缺少哪种？
   - 如果不能，需要新增哪种实体？
3. 统计覆盖率：充分描述 / 总查询数
4. 目标：覆盖率 ≥ 80%。低于此值 → 补充实体类型后再上线
```

**上线后（持续监控）：**
```python
# 每条查询记录日志
query_log = {
    "user_input": "为什么用户登录突然变慢了？",
    "extracted_entities": {"module": "用户登录"},     # 只抽到1个实体
    "completeness": "bare",                            # 实体极稀疏
    "retrieval_strategy": "semantic_only",             # 降级为纯语义搜索
    "user_feedback": None,                             # 等用户反馈
    "query_type_inferred": "root_cause_analysis"       # 事后人工标注
}
```

每周 review：
1. 筛选 `completeness == "bare"` 的查询（预计占 15%）
2. 人工归因——是因为用户问了模糊问题（合理 bare），还是因为实体类型不够用（需要扩展 schema）
3. 如果同一类"实体覆盖不到"的查询反复出现 → 新增实体类型

**新增实体的门槛：** 同一类未覆盖查询一周内出现 ≥ 10 次。不够这个频率的，不值得为它扩展 schema 增加 LLM 抽取复杂度。

##### 实体不覆盖 ≠ 系统失败

这是最关键的设计原则：**实体是检索加速器，不是检索闸门。** 即使一条查询匹配零个实体，系统仍然可以工作：

```
实体抽取失败 → bare 处置 → 原始 query 直接做向量检索 → 返回结果
                                                     │
                                               标注"结果基于语义搜索"
```

用户感知到的是"结果不够精准"，而不是"系统拒绝回答"。两者的用户体验差距巨大——前者让用户下次学会提供更多信息（"哦，我下次带需求 ID 来"），后者让用户不再使用。

**真正需要恐慌的不是实体覆盖不到，而是用户想查的信息你的三源数据里根本没有。** 这种 case 不是实体设计能解决的，要在 Shadowing 阶段就暴露出来——如果 20% 以上的用户问题无法从 PRD+代码+SQL 三者中任何一个找到答案，说明数据源的选择本身有问题，或者需要引入第四种数据源（比如 Jira、Confluence 非 PRD 页面、监控系统）。

#### 反事实推演：去掉实体抽取会怎样

这个问题的价值在于——如果去掉实体抽取影响不大，那整个抽取模块就是过度设计。反过来，如果影响显著，那投入就是值的。逐 Worker 分析。

##### Doc Worker：影响中等

去掉实体抽取后，Doc Worker 的检索从两路变成一路：

```
有实体:  req_ids → 元数据精确过滤 (WHERE req_id='REQ-187') ✓ 100% 精准
         module  → 语义搜索 "用户登录 PRD 需求"            ✓ 缩小到功能域
         
无实体:  原始query → 纯语义搜索                              ⚠ 全靠embedding质量
```

**真正损失的场景：需求 ID 精确查询。** "REQ-187" 这种短字符串，BGE-M3 的嵌入向量跟文档分块里的 "REQ-2024-0187" 的语义相似度不保证能排进 Top-K。因为 "REQ-187" 和 "REQ-2024-0187" 是不同的 token 序列，模型可能理解不了它们是同一个东西——它不像人类看到 "REQ-187" 就知道去搜完整编号。

**其余场景影响不大。** "用户登录的 PRD 文档" 这类 module 级查询，纯语义搜索本来就能搞定。`time_range` 和 `doc_types` 只是锦上添花的过滤条件。

> **量化估计：** 去掉实体后，Doc Worker 的 Recall@10 下降约 10-15 个百分点（主要来自 req_ids 精确匹配的损失）。

##### Code Worker：影响中等偏高

```
有实体:  code_refs → grep "TokenService" + 文件名 "oauth.py"  ✓ 毫秒级精确
         req_ids   → git log --grep "REQ-187"                   ✓ 精准关联
         module    → 语义搜索 code embedding                     缩小范围
         
无实体:  原始query → 纯代码嵌入向量检索                           ⚠ 
```

**代码嵌入的质量是最大的不确定性。** BGE-M3 是在自然语言上训练的，不是代码。中文代码库（中文注释、中文变量名）的嵌入质量没有经过验证。"用户登录模块的认证逻辑" 这种 query 跟 `def login_oauth(self, token: str)` 之间的语义相似度，不一定能超过跟 `def user_register(self, ...)` 的相似度。

**而且代码检索有一个特殊问题：用户要的不是"语义上相似的代码"，而是"那一份确切的代码"。** "oauth.py 里的 TokenService"——用户知道文件在哪，只是不想手动翻。有 `code_refs`，直接 `grep`；没有，就得祈祷嵌入模型的天线恰好对准了。

> **量化估计：** 去掉实体后，Code Worker 的 Recall@10 下降约 15-25 个百分点。损失最大的是"知道文件名/函数名"这类精确查询。

##### SQL Worker：影响最大——去掉实体近似于盲搜

这是三个 Worker 里退化最严重的。原因：**数据库 Schema 的语义信息极度稀疏。** 

一个典型的列名：`usr_auth_st` → 代表 `user_auth_status` → 业务含义是"用户认证状态（0=未认证 1=已认证 2=过期）"。BGE-M3 的嵌入向量看到的只是 11 个字符 `u-s-r-_-a-u-t-h-_-s-t`，它猜不到这跟"用户登录"有什么关系。

```
有实体:  table_names=["users"] → 跳过 Schema RAG，直接查 DDL         ✓ 100% 精准
         column_names=["email"] → 在表中精确定位列                     列级精准
         metrics=["订单数"]     → 决定 SELECT COUNT(order_id)         聚合行为确定
         time_range="上周"      → WHERE created_at >= NOW()-7d        时间范围确定
         group_by="按状态"      → GROUP BY status                    分组列确定
         
无实体:  原始query → Schema RAG 语义搜索                                ⚠ 极高不确定性
```

流程对比：

```
有实体: "过去7天各状态的订单数"
  → table_names=["orders"] → SELECT column_name FROM information_schema WHERE table_name='orders'
  → 拿到全部列名和注释 → metrics="订单数" → LLM 生成: SELECT status, COUNT(*) FROM orders WHERE created_at >= NOW()-7d GROUP BY status
  → 路径: 确定性查找 → 精确SQL生成
         [延迟 ~800ms, 准确率 ~85%]

无实体: "过去7天各状态的订单数"
  → 整句 embedding → Schema RAG 搜索 → 可能返回: ["orders", "order_items", "order_logs", "user_orders_view"]
  → LLM 从四个候选表中选出正确的 → 生成 SQL → SQL Guard 校验 → 失败 → 自修复 → ...
  → 路径: 模糊检索 → 猜测表 → 可能选错表 → 自修复兜底
         [延迟 ~2500ms, 准确率 ~55%]
```

**核心差异不是延迟，而是"表名选错"这个失败模式。** Schema RAG 用"过去7天各状态的订单数"去搜，返回的前 5 个表里如果 `orders` 排第二而不是第一，LLM 可能选了 `order_items` 来生成 SQL。SQL Guard 能校验语法，但校验不了"你选错表了"——`SELECT status, COUNT(*) FROM order_items` 语法完全正确。

> **量化估计：** 去掉实体后，SQL Worker 的端到端 Execution Accuracy 从 ~80% 下降到 ~50-55%。不是语义搜索不够好，而是数据库 schema 根本没提供足够的语义信号让搜索工作。

##### 综合影响

| Worker | 当前 Recall@10 | 去实体后 | 下降 | 主要损失来源 |
|--------|--------------|---------|------|------------|
| Doc | ~0.88 | ~0.75 | -13pp | req_ids 精确匹配失效 |
| Code | ~0.80 | ~0.60 | -20pp | code_refs 精确grep失效 + 代码embedding质量不确定 |
| SQL | ~0.85 | ~0.85（Recall） / ~0.50（Accuracy） | Recall不降但准确率暴跌 | 语义搜索能"找到"表但找错表；Recall 虚高 |

**注意 SQL Worker 的陷阱：** 去掉实体后 Recall 可能不降——语义搜索总能返回 N 个相关的表，里面很可能包含正确的那个。但 LLM 从 N 个候选表中选对一个的准确率，跟只有 1 个确定表完全不同。Recall 看起来还行，用户拿到的答案却是错的。这是最危险的退化模式——指标看起来 OK，用户体验已经崩了。

##### 结论：哪些实体是真正不可替代的

不是所有实体都同等重要。三个"权重 10"的精确匹配级实体——`req_ids`、`table_names`、`code_refs`——构成了系统的**确定性检索路径**。它们的共同特征：用户明确知道目标，系统应该直接定位，不需要猜测。去掉它们，系统仍然能工作（语义搜索兜底），但从"80% 情况下一次命中"变成"60% 情况下需要用户多次追问才能找到"。

三个"权重 5"的强语义级实体——`module`、`metrics`、`column_names`——是重要的检索加速器，但去掉后纯语义搜索可以部分替代。

其余辅助过滤级实体去掉后对召回率几乎没有可测量的影响，它们的价值在于提升排序精度（让对的结果排在更前面）。

**如果工程资源极度紧张被迫砍功能，保留三个"权重 10"实体 + `module`，其余的可以不做。** 这四个字段覆盖了 ~70% 的查询场景，其余场景纯语义搜索兜底。但你从一开始就说了要企业级完整方案，所以这个问题更多是"理解每个组件的边际贡献"，而不是真的砍掉。

### 查询改写设计

这是 Query Understanding 的第三个子模块。用户说的话跟系统里存的东西之间存在**词汇鸿沟**——用户说"登录不了"，PRD 里写的是"认证失败"；用户说"用户表"，数据库里是 `users`。查询改写的作用是桥接这个鸿沟。

六种改写方案，解决的问题不同，代价也不同。不是全要——按 Phase 分阶段引入，每种有明确的启用条件。

#### 方案总览

```
用户原始 query
      │
      ▼
┌──────────────────────────────────────────────────────┐
│               查询改写流水线（可插拔）                   │
│                                                      │
│  [1] 标准化 ──→ [2] 扩展 ──→ [3] 分解 ──→ [4] HyDE  │
│   (~1ms)        (~5ms)       (~800ms)     (~1500ms)  │
│                                                      │
│  从左到右：代价递增，收益也递增                           │
│  标准化和扩展始终开启；分解和 HyDE 按条件触发              │
└──────────────────────────────────────────────────────┘
      │
      ▼
  改写后的 query → 下发各 Worker
```

#### 方案 1：标准化（Normalization）— 始终开启

**解决的问题：** 用户用语和系统内部名不一致。用户的"用户表"在数据库里叫 `users`，用户的"登录不了"在 PRD 里是"认证失败"。

**实现：** 一个同义词映射表，不调用 LLM。毫秒级。

```python
SYNONYM_MAP = {
    # 表名映射
    "用户表": "users",
    "订单表": "orders", 
    "商品表": "products",
    
    # 术语映射
    "登录不了": "登录失败 OR 认证失败 OR 认证异常",
    "挂了": "服务不可用 OR 系统异常 OR 宕机",
    "慢": "性能下降 OR 响应延迟 OR 超时",
    "改需求": "需求变更 OR PRD更新",
    
    # 模块简称 → 全称
    "登录": "用户登录认证",
    "支付": "支付网关",
    "审批": "审批流程",
    
    # 缩写 → 全称（SQL Schema 场景）
    "uid": "user_id",
    "crt_tm": "created_at",
}

def normalize(query: str) -> str:
    """纯规则：遍历映射表，匹配就替换"""
    result = query
    for slang, canonical in SYNONYM_MAP.items():
        if slang in result:
            # 保留原始词 + 追加规范词（OR 连接）
            result = result.replace(slang, f"({slang} OR {canonical})")
    return result
```

**输入→输出示例：**
```
"用户表最近一周新增了多少数据？"
→ "(用户表 OR users)最近一周新增了多少数据？"

"登录挂了"
→ "(登录 OR 用户登录认证)(挂了 OR 服务不可用 OR 系统异常 OR 宕机)"
```

**维护方式：** 分三个阶段演进——人工种子→自动发现→人工审核闭环。

##### 阶段 1：冷启动——人工种子（上线前）

从数据源盘点中一次性提取。不靠猜测，从三个源的元数据中直接捞：

```
数据来源                          提取的映射项
──────────────────────────────────────────────────
数据库 information_schema     →  表名 ←→ 表注释（"users ←→ 用户信息表"）
  - table_name                 →  列名 ←→ 列注释（"usr_auth_st ←→ 用户认证状态"）
  - table_comment / column_comment
  
PRD 文档标题/目录              →  模块名 ←→ 需求ID（"用户登录 ←→ REQ-2024-0187"）
  - 文档标题层级
  - 标签/分类
  
Git 仓库目录结构               →  目录名 ←→ 模块名（"src/auth/ ←→ 认证模块"）
  - 一级/二级目录名
  
人工补充                       →  常见口语/缩写（"挂了 ←→ 服务不可用"）
  - 从 Shadowing 观察中收集
  - 团队内部黑话
```

预计冷启动可自动生成 **60-80条映射**（表名+列注释+模块名），人工补充 **20-30条**（口语/缩写）。总量 80-110 条，覆盖最常见的查询。

##### 阶段 2：自动发现——三条管道持续挖掘（上线后）

不靠人主动汇报，靠系统从查询日志中自动嗅探。

**管道 A：用户自修正挖掘（Self-Correction Mining）**

这是最强的信号。用户第一次查询没得到好结果，主动换了说法再查一遍——第二次的说法往往就是第一次的同义词。

```python
def detect_self_correction(session_logs: list) -> list[SynonymPair]:
    """
    检测"用户自己修正查询"的模式。
    
    信号规则：
    - 同一会话内，两次查询间隔 < 60 秒
    - 第一次查询 用户无点击/无追问/快速离开
    - 第二次查询 用户有正向交互（点击引用、追问细节、停留>30秒）
    - 两次查询的语义相似度 > 0.6（说明是同一意图）
    
    满足上述条件 → 提取第二次查询中新增的关键词 → 
    这些词可能是第一次查询的"正确说法"
    """
    pairs = []
    for session in group_by_session(session_logs):
        for q1, q2 in sliding_window(session.queries, gap_seconds=60):
            if is_low_engagement(q1) and is_high_engagement(q2):
                if semantic_similarity(q1.text, q2.text) > 0.6:
                    new_terms = extract_new_keywords(q2.text, q1.text)
                    old_terms = extract_keywords(q1.text)
                    for new, old in zip(new_terms, old_terms):
                        pairs.append(SynonymPair(
                            user_said=old,      # 用户第一次说的（不好的表述）
                            better_match=new,   # 用户第二次说的（检索到了结果的表述）
                            source="self_correction",
                            confidence=0.7      # 用户行为信号，高置信度
                        ))
    return pairs
```

**管道 B：检索落差分析（Retrieval Gap Analysis）**

```python
def detect_retrieval_gap(query_log: QueryLog) -> list[SynonymPair]:
    """
    分析"改了检索词后检索质量跳跃"的 case。
    
    信号规则：
    - 意图分类相同（同一个用户意图）
    - 两次查询的检索 Recall@10 差异 > 0.3
    - 高 Recall 查询中有低 Recall 查询中不存在的关键词
    
    → 这些新增的关键词就是"让搜索变好的词"
    → 低 Recall 查询中的对应词可能需要映射到这些词
    """
```

**管道 C：LLM 批量挖掘（月度离线任务）**

每个月跑一次离线任务，让 LLM 扫描所有查询日志，找潜在的映射对：

```python
LLM_MINING_PROMPT = """
你是一个术语挖掘专家。以下是本月用户查询中"检索结果不理想"的查询列表（用户点击率低、无追问、快速离开）。

为每个查询建议：哪些词替换成哪些词后，检索结果可能会更好？

输入格式（每行一条）：
{query_text} | extracted_entities | retrieval_score

输出 JSON:
[{"original_term": "挂了", "suggested_canonical": "服务不可用", "reason": "口语→技术术语", "confidence": 0.9}]

只输出 confidence >= 0.7 的建议。
"""
```

**管道对比：**

| 管道 | 触发频率 | 信号强度 | 误报率 | 适用场景 |
|------|---------|---------|-------|---------|
| A: 用户自修正 | 实时（每次会话结束） | 极高——用户自己验证了正确说法 | 低 | 口语→术语映射 |
| B: 检索落差 | 每日批量 | 高——数据驱动 | 中 | 新术语、新缩写 |
| C: LLM 挖掘 | 月度离线 | 中——无用户验证 | 中高 | 长尾、低频词的发现 |

##### 阶段 3：人工审核闭环

自动发现不是自动上线。每条候选映射都进审核队列：

```
候选映射（置信度 0.7-0.9）──→ 人工审核队列（Web UI，非代码仓库）
                               ├─ ✓ 批准 → 立即生效（写入映射表 + 记录审批人）
                               ├─ ✗ 拒绝 → 丢弃（记录理由，避免后续重复发现）
                               └─ ↻ 修改 → 调整映射关系后批准

候选映射（置信度 ≥ 0.9）  ──→ 自动生效（静默） + 每周汇总邮件通知
                               高置信度、低风险，不需要人工干预
```

**审核界面的信息：**

```
│  - 同一用户 2026-06-03 14:23: "支付服务异常"       │
```

##### 防止映射表腐化

**问题：** 映射表只增不减，一年后可能膨胀到 500+ 条，其中很多已经过时（表被删了、模块改名了、术语不流行了）。

**对策：**

```python
def decay_check(mapping: SynonymPair) -> bool:
    """
    每条映射定期做"衰变检查"：
    
    1. 该映射近 30 天被触发过吗？  没有 → 标记为"低频"，需要复查
    2. 映射目标还存在吗？          表被删了/PRD被归档了 → 标记为"失效"
    3. 用户还在用原词吗？          原词近 30 天零出现 → 标记为"过时"
    
    每月自动运行 → 生成"建议清理"列表 → 人工确认后删除
    """
```

**映射表健康度仪表盘：**
- 总映射数（目标：100-300 条，超出上限意味着需要清理）
- 本月新增 / 本月清理
- 映射命中率（有映射触发的查询 / 总查询——如果映射表一直在涨但命中率不涨，说明加了很多无用映射）
- Top 10 最高频触发映射（哪些口语最常被改写）

##### 为什么不全自动化

三个原因：
- **误报会把用户带到错误方向。** "支付挂了"如果自动映射到"薅羊毛攻击"（LLM 的联想跳跃），用户搜"支付挂了"返回一堆反欺诈代码，完全不可用。
- **映射表是代码仓库外的热配置，变更零风险。** 加一条映射不需要重启服务，不需要 CI/CD。但这也意味着没有测试保护——人工审核是唯一的门禁。
- **审核本身就是信号。** 如果某个映射被反复提交但审核人都拒绝了，说明这里存在系统性混淆，可能需要从数据源头解决（比如让 DBA 给表加注释），而不是在改写层打补丁。

**适用 Worker：** 全部。成本接近于零，始终开启。

---

#### 方案 2：扩展（Query Expansion）— 始终开启

**解决的问题：** 用户查询太短，缺少上下文词。BM25 关键词匹配和向量检索在短查询上的表现都差，因为信息量不足。

**实现：** LLM 生成 3-5 个相关词/短语，追加到原始查询。不改变查询语义，只补充词汇。

```python
EXPANSION_PROMPT = """
为以下用户查询生成 3-5 个相关的搜索关键词或术语（仅输出关键词列表，用逗号分隔）。
不要改变查询的本意，只是为了提高搜索召回率而补充同义词和相关术语。

查询: {query}

关键词:"""

# 输出: "OAuth2.0, 第三方登录, SSO, token刷新, 认证授权"
```

**输入→输出示例：**
```
"用户登录怎么做的？"
→ "用户登录怎么做的？ OAuth2.0 SSO 认证授权 token验证 登录流程"
```

**代价：** 一次轻量 LLM 调用（Haiku，~300ms），可选缓存（相同 query 24h 内复用扩展结果）。

**适用 Worker：** Doc Worker 和 Code Worker 收益最大（短 query 匹配长文档/代码）；SQL Worker 有标准化就够，扩展容易引入不相关术语干扰 Schema 检索。

**风险：** 扩展过度会引入噪声。对策：限制扩展词数量 ≤ 5，且只在原始 query ≤ 30 字时触发——长查询本身信息量够。

---

#### 方案 3：分解（Multi-Query Decomposition）— 条件触发

**解决的问题：** 一个查询里包含多个独立子问题，或者跨源查询时不同源需要不同的检索词。直接用一个 query 同时搜三种源，效果差。

**触发条件：** `is_cross_source == true` 或用户查询中包含"和"、"以及"、"还有"等多问题连接词。

**实现：** LLM 将复杂查询分解为 2-4 个独立子查询。

```python
DECOMPOSE_PROMPT = """
将以下复杂查询分解为 2-4 个独立的子查询，每个子查询针对一个特定的数据源或子问题。

数据源:
- doc: PRD文档、产品需求
- code: 代码实现  
- sql: 数据库查询/数据统计

查询: {query}

以 JSON 输出: [{"query": "子查询1", "target": "doc"}, ...]

示例:
"REQ-187改了哪些功能，对应的代码在哪，影响了哪些表？"
→ [
  {"query": "REQ-187 PRD需求变更内容 改了什么功能", "target": "doc"},
  {"query": "REQ-187 相关代码文件 代码变更", "target": "code"},
  {"query": "数据库表 受REQ-187影响 相关数据表", "target": "sql"}
]
"""
```

**输入→输出示例：**
```
"用户登录的PRD内容，以及登录相关的代码和用户表结构"

→ Doc Worker 收到: "用户登录 PRD需求 功能规格 登录流程"
→ Code Worker 收到: "用户登录代码实现 auth login 认证逻辑"
→ SQL Worker 收到: "users表结构 DDL 用户表字段"
```

**代价：** 一次 LLM 调用（Sonnet/Haiku，~500ms），只在跨源查询时触发（预计 ~30% 流量）。

**不触发的情况：** 单源简单查询直接跳过分解，原始 query 经过标准化+扩展就够了。

**与 Supervisor 路由的关系：** 分解在意图分类之后执行——分类先确定需要哪些源，分解再为每个源生成定制化检索词。分类结果决定"查几个源"，分解结果决定"每个源具体查什么词"。

---

#### 方案 4：HyDE（Hypothetical Document Embeddings）— 条件触发

**解决的问题：** 这是向量检索领域最经典的"冷启动"问题——用户的短 query 和系统中长文档的嵌入向量不在同一个语义空间里。用户问 8 个字"用户登录怎么做的"，文档分块 500 个 token。两者的 embedding 相似度天然偏低。

**HyDE 的思路：** 让 LLM 先根据用户 query 生成一段假设性答案（hypothetical document），然后用这段假设性答案的 embedding 去检索。假设性答案的风格和长度更接近真实文档，embedding 匹配度更高。

```
传统方式:  "用户登录怎么做的" ──embedding──→ [0.12, -0.34, ...] ──相似度──→ 文档分块
                                                      ↑
                                                 信息量差距大，相似度低

HyDE:      "用户登录怎么做的" ──LLM──→ "用户登录功能通过OAuth2.0协议实现，
                                       用户点击登录按钮后跳转至授权页面..." 
                                              │
                                              ──embedding──→ [0.15, -0.41, ...] ──相似度──→ 文档分块
                                                                    ↑
                                                              风格和长度接近文档，相似度更高
```

**实现：**

```python
HYDE_PROMPT = """
你是一个技术文档助手。根据用户的问题，写一段可能出现在企业技术文档中的回答（约200字）。
不要追求完全准确——这只是用来搜索真实文档的"诱饵"。风格模仿PRD文档或技术方案。

用户问题: {query}

假设性回答:"""

def hyde_rewrite(query: str) -> str:
    hypothetical_doc = llm.generate(HYDE_PROMPT.format(query=query))
    # 用假设性文档的 embedding 替代原始 query 的 embedding 做检索
    return hypothetical_doc
```

**触发条件：** 不是所有查询都需要。只在以下条件同时满足时触发：

1. 原始 query ≤ 30 字（短 query 才有冷启动问题）
2. `completeness` 评估为 "partial" 或 "bare"（没有精确匹配实体兜底）
3. 目标 Worker 是 Doc Worker 或 Code Worker（SQL Worker 的 Schema RAG 不需要 HyDE，因为检索对象是短 DDL 语句）

**代价：** 一次 LLM 生成（~1500ms），但 HyDE 的延迟和检索延迟可以并行——HyDE 生成的同时原始 query 的检索也可以先跑，HyDE 的结果作为补充检索而非替代。

```python
# 并行策略：原始query检索和HyDE检索同时发
results = await asyncio.gather(
    vector_search(original_query),    # 原始检索
    vector_search(hyde_document),     # HyDE检索（如果触发）
)
merged = rrf_fusion(results)  # RRF合并两路结果
```

**适用 Worker：** Doc Worker（收益最大，PRD 文档长而 query 短）。Code Worker 次之。SQL Worker 不需要。

**风险：** HyDE 生成的假设性文档可能"脑补"错误信息，引导检索走向错误方向。对策：始终同时保留原始 query 的检索结果，HyDE 只作为补充信号，不替代原始检索。

---

#### 方案 5：退一步改写（Step-Back Prompting）— Phase 3+

**解决的问题：** 用户问了一个高度具体的问题，但这个具体问题需要更广泛的背景知识才能正确回答。

```
用户: "oauth.py第42行为什么用refresh_token而不是access_token？"

直接搜: "oauth.py第42行 refresh_token access_token" → 找不到解释"为什么"
退一步: "OAuth token刷新机制的设计原理" → 找到背景知识
         再回答: "因为refresh_token有效期更长，存在服务端更安全..."
```

**触发条件：** 查询包含"为什么"、"原因"、"设计思路"等归因词，且 code_refs 指向了具体文件/行号。

**代价：** 两次 LLM 调用（退一步生成 + 正常检索），延迟增加 ~2000ms。Phase 3 再引入，v1 不实现。

---

#### 方案 6：上下文感知改写（Context-Aware Rewriting）— Phase 3+

**解决的问题：** 多轮对话中，用户的追问省略了上下文。"那个功能呢？"、"它依赖哪些表？"——"那个"、"它"需要通过对话历史来消解。

**触发条件：** 查询包含指代词（"它"、"这个"、"那个"）且对话历史非空。

**实现：** 从上一轮分类和实体中取出上下文，替换指代词。

```python
# 上一轮: user="用户登录的实现", entities.module="用户登录", req_ids=["REQ-187"]
# 本轮: "它依赖哪些数据库表？"
# 改写后: "用户登录功能依赖哪些数据库表？"
```

**代价：** 本身不增加 LLM 调用（对话历史已在上下文窗口内），但需要维护会话状态。Phase 3+ 实现。

#### 改写流水线的执行顺序和 Phase 规划

```
                    标准化              扩展               分解              HyDE
用户query ──→  [始终开启]  ──→  [始终开启]  ──→  [跨源时触发]  ──→  [短query+无实体]
               ~1ms            ~300ms           ~500ms           ~1500ms
                 │                │                │                │
                 └────────────────┴────────────────┴────────────────┘
                                    │
                                    ▼
                            改写后的 query 集合
                            (1-N 个，取决于是否分解)
                                    │
                            ┌───────┼───────┐
                            ▼       ▼       ▼
                         Doc W   Code W   SQL W
```

| Phase | 引入方案 | 理由 |
|-------|---------|------|
| Phase 1（SQL Worker） | 标准化 + 扩展 | 标准化解决"用户表→users"映射；扩展补充查询词汇。这两个是 Text-to-SQL 准确率的基础保障 |
| Phase 2（Doc Worker） | HyDE | PRD 文档长而 query 短，HyDE 的收益最大。延迟问题通过并行检索解决 |
| Phase 2 | 分解 | 跨源查询开始出现（Doc + SQL），分解为两源生成各自的检索词 |
| Phase 3 | 退一步改写 | 用户开始问"为什么"类归因问题，需要背景知识检索 |
| Phase 3 | 上下文感知改写 | 多轮对话场景成熟，用户习惯追问 |

**每个 Phase 引入改写方案前后的 A/B 对比指标：** Recall@10 变化 + P99 延迟变化。如果新增改写方案延迟增加 > 50% 但 Recall 提升 < 5pp，回滚该方案——不值得。

### 数据摄入管道（离线/异步）

```
PRD 文档 (Confluence/Wiki)
  → Docling/Unstructured 解析
  → 递归语义分块（按段落+标题自然边界切割，目标 ~500 tokens/块，50-token overlap；使用 tiktoken cl100k_base tokenizer）
  → BGE-M3 嵌入 → PGVector
  → 触发方式：Webhook（Confluence 页面更新事件）或定时全量同步（每日凌晨）

代码仓库 (Git)
  → TreeSitter AST 解析（保留函数/类边界，按语法单元分块）
  → 元数据提取（文件路径、函数签名、imports、commit message 中的需求 ID 如 [REQ-1234]）
  → BGE-M3 嵌入 → PGVector
  → 触发方式：Git Webhook（push 事件）→ 增量索引变更文件；存量代码全量索引

SQL 数据库 (PostgreSQL/MySQL)
  → Schema 自省（表结构、列类型、外键关系、列注释作为业务语义描述）
  → DDL + pg_stat_statements 采样（Top 100 高频查询作为 few-shot 示例；需 DBA 配合开通 pg_stat_statements 扩展）
  → BGE-M3 嵌入 → PGVector
  → 触发方式：定时轮询 information_schema（10min 间隔）+ 手动触发刷新（DDL 变更后）
```

### Code Worker 的双引擎检索：为什么 AST 和嵌入模型缺一不可

Code Worker 是三个 Worker 里唯一同时使用结构化解析（TreeSitter AST）和语义嵌入（BGE-M3）的。这不是冗余——它们解决的是两种完全不同的搜索问题。

#### 两种搜索原语

| | AST 结构化搜索 | BGE-M3 语义搜索 |
|---|---|---|
| **搜什么** | 代码的结构：函数名、类名、调用关系、import 链 | 代码的行为：这段代码做了什么 |
| **怎么搜** | 精确匹配 + 图遍历 | 向量相似度 |
| **典型问题** | "oauth.py 的 `token_refresh` 函数" | "用户登录的认证逻辑在哪" |
| **查询特征** | 用户知道精确的名字 | 用户描述行为，不知道函数名 |
| **延迟** | ~10ms（grep/图查询） | ~50ms（向量检索） |

#### 为什么不能只用 AST

AST 能搜到代码的**名字**，搜不到代码的**意图**。

```
用户: "处理OAuth token过期的逻辑在哪？"

AST方法: 
  grep "OAuth" → 零结果（代码里写的是 refresh_token, access_token_expiry）
  grep "token过期" → 零结果（代码用英文写的: token_expiry, TTL）
  grep "过期" → 只有注释里有 // 处理过期 → 找到了，但漏了 token_refresh() 里隐含的过期逻辑

AST结果: 不全。因为用户用的业务术语（"过期"）不等于代码里的技术术语（expiry/TTL）。
```

#### 为什么不能只用嵌入模型

嵌入模型能搜到**语义相似**的代码，但搜不到用户**明确指定**的精确目标。

```
用户: "oauth.py 第42行的 token_refresh 为什么用 refresh_token 而不是 access_token？"

嵌入方法:
  BGE-M3("oauth.py token_refresh refresh_token access_token") → 
  Top-5 结果: 
    1. auth.py 的 refresh_credentials()    ← 语义相似，但不是用户要的
    2. oauth.py 的 login_oauth()          ← 同文件，但不同函数
    3. token_service.py 的 rotate_token() ← 同概念，不同文件
    4. oauth.py 的 token_refresh()        ← 这个才对！
    5. session.py 的 extend_session()     ← 不相关

AST方法:
  grep "token_refresh" → 文件名 "oauth.py" + 函数名 → 0.01秒精确定位到第42行
  
AST结果: 精准。因为用户给了精确的名字——文件名+函数名——不需要猜。
```

#### 关键区分：用户查询的两类模式

```python
def route_code_search(entities: ExtractedEntities, user_query: str) -> SearchStrategy:
    """
    按实体类型决定走哪条搜索路径。
    """
    has_exact_refs = bool(entities.code_refs)  # 用户给了具体的文件名/函数名/类名
    
    if has_exact_refs:
        # 路径A: 精确匹配优先
        # AST grep 文件名+函数名 → 毫秒级定位
        # 嵌入搜索作为补充（搜相关代码，用于"这个函数还调用了哪些相关函数"）
        return SearchStrategy(
            primary="AST_EXACT",
            secondary="EMBEDDING_CONTEXT",  # 仅用于扩展上下文
            ast_query=grep_query(entities.code_refs),
            embedding_query=f"{entities.module} {user_query}"  # 用模块语义补充上下文
        )
    else:
        # 路径B: 纯语义搜索
        # 用户没有明确指出文件/函数→只能用自然语言描述行为
        # AST在这里帮不上忙——没有具体名字，图遍历无从下手
        return SearchStrategy(
            primary="EMBEDDING_SEMANTIC",
            secondary=None,
            embedding_query=user_query  # 改写后的完整query
        )
```

**触发精确路径（AST primary）的条件：** `code_refs` 非空。占总查询约 30%。

**触发语义路径（Embedding primary）的条件：** `code_refs` 为空，只有 `module` 或纯自然语言。占总查询约 70%。

#### 两层如何协作

精确路径和语义路径不是"二选一"，而是主次配合：

```
用户: "TokenService 里怎么处理 refresh_token 的？"

1. AST 精确: grep "TokenService" → 找到类定义
   grep "refresh_token" → 找到类中使用 refresh_token 的所有方法
   → 返回: token_refresh(), validate_token(), rotate_credentials()
   延迟: ~10ms

2. 嵌入扩展: 以上三个方法的调用者 + 被调用者
   用 BGE-M3 搜"token refresh rotation expiry" → 找到相关代码
   → 返回: on_token_expired() 事件处理器, TokenStorage 类, refresh_token 写入的数据库表
   延迟: ~50ms

3. 结果合并: AST 精确结果（核心）+ 嵌入扩展结果（上下文）→ 传给 LLM 生成回答
```

**AST 回答"在哪里"和"是什么"，嵌入模型回答"还跟什么相关"。** 只有前者，用户拿到一个孤立的函数，看不到它跟系统其他部分的关系。只有后者，用户拿到一堆语义相似但可能不精确的代码片段。

#### 对中文代码库的特殊意义

BGE-M3 虽然在自然语言上表现好，但**中文代码库的语义检索是未经验证的领域**。

```
用户: "用户登录验证"
代码: def authenticate_user(credentials): ...
```

中文 query 和英文代码之间没有词汇重叠。在没有 AST 精确匹配的 70% 查询中，系统全靠 BGE-M3 的跨语言 embedding 质量。如果 BGE-M3 的中→英代码映射效果差，Code Worker 在这 70% 查询上基本是半盲的。

**这就是为什么在 Open Questions 第 5 条中把"中文代码嵌入质量评估"列为 Phase 3 前的硬性验证项**——如果 BGE-M3 的中文代码检索不达标，需要换用多语言代码嵌入模型（如 UniXcoder）或对 BGE-M3 做代码语料微调。

#### 代码 embedding 到底存了什么

直接回答：**是的，代码经过 AST 分块后嵌入为向量存入 PGVector。** 但"存什么内容"比"存不存"更重要。

##### 分块单元：函数/类，不是 token 数

文档分块按段落边界（~500 tokens），代码不一样——代码的结构单元是函数和类，强行按 token 切割会把一个函数切成两半，embedding 质量断崖式下跌。

```
文档分块:  ┌──── 段落1 ────┐┌──── 段落2 ────┐┌──── 段落3 ────┐
代码分块:  ┌── def login() ──┐┌── class TokenService ──────────┐
                        按函数/类边界，长度不固定
```

TreeSitter 的作用：不参与检索，只做**语法感知的边界识别**——"这个函数从第 42 行开始，第 87 行结束，包括它的 docstring、函数体、内部的所有逻辑"。每个函数/类作为一个独立的 embedding chunk。

##### 每条 chunk 存两种东西

```sql
CREATE TABLE code_chunks (
    id UUID PRIMARY KEY,
    
    -- 用于语义检索（BGE-M3 嵌入）
    content TEXT,                -- 函数/类的完整源代码（含注释和docstring）
    embedding VECTOR(1024),      -- BGE-M3(content) 的嵌入向量
    
    -- 用于精确检索（AST，不需要向量）
    file_path TEXT,              -- src/auth/oauth.py
    function_name TEXT,          -- token_refresh
    class_name TEXT,             -- TokenService
    language TEXT,               -- python
    imports TEXT[],              -- ["from jose import jwt", "import redis"]
    calls TEXT[],                -- 被调用的函数: ["jwt.decode", "redis.get"]
    called_by TEXT[],            -- 调用者: ["login_oauth", "validate_session"]
    
    -- 用于跨源关联
    req_ids TEXT[],              -- 在git log中关联的需求ID: ["REQ-187"]
    commit_hash TEXT,            -- 最后修改的commit
    author TEXT,                 -- 最后修改人
    
    -- 元数据
    repo TEXT,                   -- 所属仓库
    branch TEXT,                 -- 分支
    updated_at TIMESTAMP
);

-- 向量索引（语义搜索用）
CREATE INDEX idx_code_embedding ON code_chunks USING hnsw (embedding vector_cosine_ops);

-- 精确搜索不用向量索引——直接走B-tree
CREATE INDEX idx_code_file ON code_chunks (file_path);
CREATE INDEX idx_code_function ON code_chunks (function_name);
CREATE INDEX idx_code_class ON code_chunks (class_name);
```

##### 为什么存源代码而不是存摘要

两种选择：

| | 存源代码（本方案） | 存 LLM 生成的摘要 |
|---|---|---|
| **怎么生成** | 直接取函数体原文 | 每个函数调一次 LLM 生成"这个函数做了什么"的自然语言描述 |
| **成本** | 零额外成本 | 10 万个函数 × 一次 LLM = 大量 token |
| **中文召回** | BGE-M3 的中→英跨语言映射，未经验证 | 摘要是中文的，跟中文 query embedding 天然匹配 |
| **返回给用户** | 直接展示源代码 + 文件路径 | 展示摘要，用户看源码需要再跳一次 |
| **维护** | Git webhook → 增量更新变更文件 | 每次代码变更都需要重新生成摘要 |
| **幻觉风险** | 零 | LLM 可能错误理解函数行为，摘要和实际代码不一致 |

**选择存源代码的原因：**

第一个原因是成本——Phase 1 不引入摘要生成，减少 LLM 调用。第二个原因是准确性——LLM 生成的摘要可能错误描述函数行为，用户看到摘要说"这就是我要的"，打开源码发现不是，信任直接崩塌。

但**摘要方案在 Phase 3 作为增强是可选的**——当 BGE-M3 的中→英代码检索被验证不够好时，用 LLM 为高频检索的函数生成中文摘要，作为附加 chunk 存入 PGVector：

```python
# Phase 3 可选增强：为无 docstring 的函数生成中文摘要
if not has_docstring(function) and retrieval_frequency(function) > threshold:
    summary = llm.generate(f"用一句话中文描述这段代码的功能:\n{function.source}")
    summary_chunk = CodeChunk(
        content=summary,     # "验证OAuth2.0 refresh_token的有效性，检查过期时间并更新token"
        function_name=function.name,
        file_path=function.file_path,
        embedding=bge_m3.embed(summary)
    )
    pgvector.insert(summary_chunk)
```

这不是替代源代码 embedding，而是在它之上叠加一层"中文语义层"，让中文 query 更容易匹配到正确的代码。只有在 BGE-M3 中文代码检索不达标的证据出现后才触发这个增强。

##### 关于调用图（calls / called_by）

`calls` 和 `called_by` 不是来自 embedding，是 AST 分析 build 出来的调用图。这两个字段存储在 PGVector 中只是为了方便在检索结果中附带展示——"这个函数被 `login_oauth` 调用，它内部调用了 `jwt.decode`"。

它们不参与向量检索（调用关系无所谓语义相似度），但在 AST 精确匹配路径中非常有用——用户搜到 `token_refresh` 后，顺藤摸瓜找到它的调用链，回答"谁调用了它"和"它影响了谁"。

#### 关键修正：代码 embedding 不应是 Code Worker 的主检索路径

上面的设计将嵌入模型定位为 70% 查询的**主检索路径**。研究证据和工程实践表明这个定位是错误的。需要修正。

##### 研究证据

**GrepRAG（ISSTA 2026）**——目前最直接的代码检索对比实验——对 Python 代码补全任务的测试结果：

| 方法 | 准确率 | 延迟 |
|------|--------|------|
| **GrepRAG（ripgrep）** | **42.29%** | **0.018s** |
| RLCoder（强化学习） | 39.46% | ~3.0s |
| VanillaRAG（BM25） | 24.99% | ~3.0s |
| GraphCoder（图结构） | 19.44% | 6.9s |

轻量级 grep 不仅比嵌入检索准确率高 17 个百分点，而且快 170 倍。这个差距在严格的结构化代码任务上尤其显著。

**Cursor 的补充数据：** 在 grep 基础上叠加语义搜索，准确率提升 6.5%-23.5%（取决于模型）。注意：语义搜索是**补充**，不是替代——两者混合比单独任何一种都好。

**Chroma 和 Google DeepMind 的研究发现了三个核心问题：**

1. **语义相似 ≠ 功能正确。** 把 `<` 改成 `<=`，嵌入向量几乎不动，但程序行为完全变了。两个变量名不同但功能相同的函数，在向量空间里离得很远。
2. **规模天花板。** 512 维嵌入在 ~50 万条文档时开始崩溃。向量空间变得拥挤，最近邻返回的噪声越来越多。在这个规模下 BM25 关键词搜索**反超**了神经嵌入。
3. **干扰项（distractor）问题。** 语义搜索返回语义相似但实际不相关的代码（handler 函数、test fixtures、废弃实现、相似命名的工具函数），这些"看起来相关但事实无关"的结果被证明对 LLM 造成的幻觉**比随机无关文本还严重**。

##### Claude Code 的做法

Claude Code 完全不做代码索引和 embedding。它的代码理解方式是：

```
1. 用 Grep 搜相关的符号/字符串 → 确定文件范围
2. 用 Read 直接读文件（不是读取 embedding chunk，是完整文件）
3. 在上下文中理解代码结构（调用关系、继承链、数据流）
4. 根据需要再 Grep → 再 Read → 再理解
```

Boris Cherny（Claude Code 技术负责人）公开说过放弃本地向量数据库的原因——**嵌入索引的过期问题**。代码一 commit，索引就跟不上。agentic grep 永远搜索的是代码的当前状态。

Claude Code 这种"不索引、直接读"的方案之所以可行，有两个前提：
- 200K token 的上下文窗口（一次性装下大量代码，不需要分块）
- 单用户、单任务的交互模式（agent 可以慢慢搜，用户等得起）

##### 这两个前提在我们的 RAG 系统里不完全成立

- 我们的系统是**多用户共享的知识库**，同一个问题可能被不同用户反复问。每次现场 grep → read → 理解是对资源的浪费。预建索引的价值在于"回答一次，后续所有人受益"。
- 用户期望的是**秒级响应**，不是 agent 探索几分钟。纯 agentic grep 在复杂查询上的延迟不可接受。
- 但 grep 比 embedding 更准确这个事实不因为场景不同而改变。

##### 修正方案：grep/AST 主检索，embedding 降级为"无 grep 命中时的备援 + 上下文扩展"

```
修正前（原设计）:
  code_refs 非空 → AST 精确（30%）
  code_refs 为空 → 嵌入语义（70%）← 这是错误的

修正后:
  所有查询 → grep/AST 先行（用改写后的 query 做关键词+结构搜索）
             ├─ 结果充足（>5条）→ 直接返回（约50%查询）
             ├─ 结果不足（1-5条）→ 嵌入补充上下文（约35%查询）
             └─ 零结果 → 纯嵌入兜底（约15%查询）
```

```python
def revised_code_search(query: str, entities: ExtractedEntities) -> SearchResult:
    """
    修正后的 Code Worker 检索策略：grep 优先，embedding 兜底。
    """
    # Step 1: grep/AST 先行（始终执行，~10ms）
    grep_results = grep_codebase(
        terms=extract_search_terms(query, entities),
        file_patterns=infer_file_patterns(entities.module),
        ast_nodes=["function", "class", "method"]
    )
    
    if len(grep_results) >= 5:
        # 情况A: grep 结果充足 → 直接使用（50%查询）
        # 不需要 embedding，grep 已经精准定位了
        return SearchResult(
            primary=grep_results,
            primary_method="grep",
            supplementary=None
        )
    
    elif len(grep_results) > 0:
        # 情况B: grep 有结果但不够 → embedding 扩展上下文（35%查询）
        # 用 grep 结果中出现的函数名/类名/import 构建语义扩展 query
        context_terms = extract_identifiers(grep_results)
        embedding_results = vector_search(
            query=f"{query} {context_terms}",
            top_k=10
        )
        return SearchResult(
            primary=grep_results,            # grep 的精确结果
            supplementary=embedding_results,  # embedding 扩展的相关代码
            primary_method="grep",
            supplementary_method="embedding"
        )
    
    else:
        # 情况C: grep 零结果 → embedding 兜底（15%查询）
        # 用户用的业务术语跟代码中的技术术语完全对不上
        # 比如用户说"限流"，代码里写的是 rate_limit / throttle
        embedding_results = vector_search(query=query, top_k=10)
        return SearchResult(
            primary=embedding_results,
            primary_method="embedding",
            supplementary=None,
            note="grep 未匹配到关键词，使用语义搜索。结果可能不够精确。"
        )
```

**这个修正改变了什么：**

| 维度 | 修正前 | 修正后 |
|------|--------|--------|
| 70% 查询的主检索方式 | 纯 embedding | grep 先行 |
| embedding 的角色 | 主引擎 | 备援 + 上下文扩展 |
| 最坏情况延迟 | ~50ms（向量检索） | ~60ms（grep 10ms + embedding 50ms 串行） |
| 最好情况延迟 | ~50ms | **~10ms**（grep 直接命中） |
| 索引过期风险 | 高（代码变了 embedding 跟不上） | 低（grep 永远搜当前状态，embedding 只做补充） |
| 对中文代码库的依赖 | 高（全靠 BGE-M3 中→英映射） | 低（grep 直接搜标识符，不依赖跨语言映射） |

##### 那 embedding 索引还要不要建？

**要。** 但不是为了 70% 的查询，而是为三个场景：

1. **grep 零结果的 15% 查询**——用户用纯业务术语描述行为，代码里没有任何关键词匹配。这是 embedding 的"长尾救急"价值。
2. **grep 结果不足时的上下文扩展**——grep 找到了目标函数，embedding 补充"还有哪些相关代码"，避免用户拿到孤立结果。
3. **跨源关联**——从 Doc Worker 的需求描述（自然语言）→ 用 embedding 找到相关的代码区域 → 再用 grep 精确定位。这是中文→英文代码的唯一桥梁。

修正后的命题不是"要不要 embedding"，而是 **"embedding 是备用引擎还是主引擎"**——答案是备用引擎。grep/AST 是主引擎。

### SQL 正确性保障：从"语法对"到"语义对"

SQL Guard 只解决了第一层——**语法正确性**。它校验了 SQL 语法、阻止了写操作、验证了表/列存在。但语法正确的 SQL 可能语义完全错误——查了不该查的表、用错了列、算错了指标。这需要更深的保护层。

#### 两个维度的正确性

```
┌─────────────────────────────────────────────────────────────┐
│  维度1: SQL 执行正确性（"SQL 本身对不对"）                      │
│  ├─ 语法正确（SQL Guard 已覆盖）                               │
│  ├─ 语义正确（查的是用户想查的东西吗？）  ← 当前设计覆盖不足       │
│  └─ 业务正确（符合业务规则和定义吗？）                          │
│                                                             │
│  维度2: 数据正确性（"查出来的数能信吗"）                         │
│  ├─ 数据源正确（查的是正确的表/正确的副本吗？）                   │
│  ├─ 数据本身正确（数据有没有质量问题？）                         │
│  ├─ 结果可解释（用户能看懂这个数字是怎么算出来的吗？）             │
│  └─ 用户理解正确（用户对结果的解读没有偏差吗？）                  │
└─────────────────────────────────────────────────────────────┘
```

#### 维度1 补充：语义正确性——SQL Guard 之上的保护层

现有 SQL Guard 做了语法校验，但"语法正确、表名正确、语义错误"是 Text-to-SQL 最高频的静默失败模式。

**保护层 A：业务元数据注入（Schema RAG 增强）**

当前 Schema RAG 只检索 DDL。这不够——DDL 告诉你列名是 `status`、类型是 `varchar`，但不告诉你业务含义。需要在检索结果中额外注入三类业务元数据：

```sql
-- 从数据库元数据中提取，存入 PGVector 的 Schema 索引中：
{
  "table": "orders",
  "column": "status",
  "type": "varchar(20)",
  
  -- 以下来自列注释/数据字典（由DBA维护或自动从代码中提取）
  "business_meaning": "订单状态",
  "enum_values": {
    "pending": "待支付",
    "paid": "已支付", 
    "cancelled": "已取消",
    "refunded": "已退款"
  },
  "business_rules": "只有 paid 状态的订单才计入营收",
  
  -- 以下来自外键关系
  "related_tables": ["order_items", "payments"],
  "common_queries": [
    "SELECT status, COUNT(*) FROM orders WHERE created_at >= ? GROUP BY status"
  ]
}
```

**这条信息如何改变 LLM 的 SQL 生成质量：**

```
没有业务元数据:
  用户: "各状态的订单数"
  LLM: SELECT status, COUNT(*) FROM orders GROUP BY status
  问题: status 可能是逻辑删除标记，用户想要的是 order_state
  
有业务元数据:
  用户: "各状态的订单数"  
  LLM 看到: status 的业务含义是"订单状态"，enum_values 是 {待支付,已支付,已取消,已退款}
  LLM 生成: SELECT status, COUNT(*) FROM orders GROUP BY status
  LLM 补充说明: "订单状态包括待支付/已支付/已取消/已退款四种"
  → 用户确认"对，就是这个" ← 即使枚举值映射错误，用户能立即发现
```

业务元数据的来源：
- 数据库列注释（`COMMENT ON COLUMN orders.status IS '订单状态: pending/paid/cancelled/refunded'`）——已有，直接提取
- 代码中的 enum 定义（`class OrderStatus(Enum): PENDING = 'pending'`）——用 AST 从代码中提取
- 数据字典/PRD 文档——实体 `column_names` 命中后在 Doc Worker 中搜索相关定义

**保护层 B：Few-Shot 示例的质量管理**

当前设计中 `pg_stat_statements` 的采样查询被用作 few-shot 示例。但这里面有一个隐蔽问题：

```sql
-- pg_stat_statements 里采样到的查询:
SELECT status, COUNT(*) FROM orders WHERE created_at > '2026-01-01' AND status != 'deleted' GROUP BY status

-- LLM 学到了什么?
-- "查订单状态时加 status != 'deleted'" ← 这是一个隐含的业务规则
-- 如果 LLM 没看到这条示例 → 生成的 SQL 少了这个过滤 → 结果包含了已删除订单 → 数据不对
```

**对策：** few-shot 示例需要人工 curator 审核，而不是全自动从 query log 中采样。审核标准：
1. 查询是否加了必要的业务过滤条件（软删除、租户隔离、时间范围）
2. 聚合逻辑是否正确（去重、排除测试数据）
3. 标注"这条查询体现了什么业务规则"

不会全量审核——Phase 1 人工 curator 选 20-30 条高频查询作为黄金示例集，之后的审核由 LLM 辅助（标记可疑示例）+ 人工确认。

**保护层 C：自修复循环的语义增强**

当前的自修复循环只把 SQLGlot 的语法错误反馈给 LLM。需要把**执行结果的统计特征**也反馈进去：

```python
def enhanced_self_healing(sql: str, error: Exception, result: QueryResult) -> str:
    """增强自修复：不只反馈错误，还反馈结果的统计异常"""
    
    feedback = []
    
    # 原有: 语法/执行错误
    if error:
        feedback.append(f"SQL执行错误: {error}")
    
    # 新增: 结果统计异常检测
    if result:
        if result.row_count == 0:
            feedback.append("查询返回0行。可能原因: 过滤条件过严、时间范围不对、表为空")
        if result.row_count > 100000:
            feedback.append(f"查询返回{result.row_count}行，可能缺少聚合或过滤条件")
        if result.has_null_columns:
            feedback.append(f"以下列存在大量NULL值: {result.null_columns}")
        if result.has_unexpected_distribution:
            # 例如 GROUP BY 某个列后，某个分组占了99%
            feedback.append(f"分组分布异常: {result.distribution_summary}")
    
    return "\n".join(feedback)
```

**保护层 D：用户确认闸门（高风险查询）**

不是所有 SQL 都需要用户确认——简单的"查一下今天的订单数"不需要。但以下情况需要在执行前展示 SQL 等用户确认：

```python
def needs_user_confirmation(sql: str, entities: ExtractedEntities) -> bool:
    """判断是否需要用户确认"""
    HIGH_RISK_PATTERNS = [
        # 涉及金额/财务指标
        (r'\b(sum|avg|count)\b.*\b(amount|price|revenue|salary|工资|金额|营收)\b', "涉及财务指标"),
        # 全表聚合（可能很慢）
        (r'COUNT\(\*\)|SUM\(.*\).*FROM.*WHERE\s*$', "全表扫描/聚合"),
        # 跨多表JOIN
        (r'JOIN.*JOIN.*JOIN', "三个以上表关联"),
        # 时间范围异常（查了10年数据）
        (r"INTERVAL\s+'?\s*(\d+)\s*(year|month)", "时间范围异常"),
    ]
    
    for pattern, reason in HIGH_RISK_PATTERNS:
        if re.search(pattern, sql, re.IGNORECASE):
            return True
    return False
```

确认时展示：
```
即将执行以下SQL:
  SELECT SUM(amount) FROM orders WHERE status = 'paid' AND created_at >= '2026-01-01'

涉及表: orders (订单表)

预计返回: 1 行
```

#### 维度2：数据正确性——查出来的数能信吗

即使 SQL 完全正确，查出来的数据也可能有问题。

**问题来源 1：数据同步延迟**

```
用户: "今天的订单数"
SQL:  SELECT COUNT(*) FROM orders WHERE created_at >= '2026-06-05'
执行在: 只读副本（可能有 2-10 秒复制延迟）
结果: 比实际少了最近几秒的订单
```

**对策：** 在结果中标注数据新鲜度：
```
查询结果: 1,247 笔订单
数据来源: orders 表（只读副本，延迟 ~3秒）
数据截止: 2026-06-05 14:32:18
```

这个信息从 `pg_last_xact_replay_timestamp()` 获取。

**问题来源 2：数据质量问题**

即使主库数据也未必干净——测试数据混入生产、NULL 值的默认处理不一致、软删除标记不统一。

**对策：** 执行后结果质量检测：

```python
def result_quality_check(result: QueryResult, sql: str) -> QualityReport:
    """对查询结果做基本的质量扫描"""
    issues = []
    
    # 检查1: 空结果
    if result.row_count == 0:
        issues.append("查询返回0行。如果这是非预期的，可能原因: (1)过滤条件过严 (2)时间范围不对 (3)表名选错")
    
    # 检查2: NULL 比例异常
    for col in result.columns:
        null_ratio = result.null_count(col) / result.row_count
        if null_ratio > 0.5:
            issues.append(f"列 {col} 的 NULL 值占比 {null_ratio:.0%}，可能影响聚合结果")
    
    # 检查3: 数值列异常值（简单统计）
    for col in result.numeric_columns:
        stats = result.basic_stats(col)
        if stats.max > stats.p99 * 10:  # 最大值是P99的10倍
            issues.append(f"列 {col} 存在极端值: max={stats.max}, p99={stats.p99}")
    
    # 检查4: 时间范围是否合理
    # 如果用户问"最近一周"但SQL查的是全部历史数据
    if "created_at" in sql.lower() or "date" in sql.lower():
        time_span = result.time_span()
        if time_span and time_span.days > 365:
            issues.append(f"查询覆盖了 {time_span.days} 天的数据，可能超出预期范围")
    
    return QualityReport(issues=issues, confidence=1.0 - len(issues)*0.2)
```

质量问题不阻塞返回——结果照样展示，但在结果下方附加质量标注，让用户自行判断。

**问题来源 3：不可复现性**

同一个问题在不同时间问可能得到不同答案（因为数据在变化）。用户第二天问"昨天的订单数"，跟当天问"今天的订单数"——完全相同的 SQL 完全不同的结果。

**对策：** 
- 查询结果带时间戳和快照标识
- 返回的 SQL 可以直接复制到数据库客户端复现（透明性）
- 高频查询结果缓存时标注"该结果基于 {timestamp} 的数据快照"

**问题来源 4：用户误解结果**

最常见的场景——用户问"上个月的营收"，系统返回了一个数，用户以为这就是最终答案。但实际上：
- 这个数包含了测试订单
- 汇率用的是下单时的汇率还是当天的汇率
- 退款订单有没有排除

系统自己不知道这些业务规则，但可以**把不确定性暴露出来**：

```
查询结果: ¥847,230.00

⚠️ 数据质量提示:
- 此查询基于 orders 表，包含所有 status='paid' 的订单
- 未排除可能存在的测试订单（user_id < 100 的订单）
- 未进行汇率换算
- 如需更精确的财务数据，请确认: (1)是否排除内部测试订单 (2)是否需要汇率换算
```

这叫 **over-communication**——宁可多标注已知的局限，不要让用户默认"系统算出来的就是对的"。

### 多级降级策略

| 级别 | 触发条件 | 降级动作 | 自动恢复 |
|------|---------|---------|---------|
| L0 全功能 | 正常 | 混合检索 + LLM 生成 | - |
| L1 LLM降级 | 主LLM超时/错误率>阈值 | 切换到备用模型（本地部署） | 主LLM健康检查通过后切回 |
| L2 检索降级 | 向量数据库不可用/P99>500ms | 切换纯BM25关键词检索 | 向量库恢复后切回 |
| L3 缓存兜底 | 后端检索大面积故障 | 返回Redis缓存的热点问答 | 后端恢复后切回 |
| L4 静态兜底 | 所有动态服务不可用 | 返回预定义FAQ+提示联系管理员 | 系统完全恢复后切回 |

### 部分失败处理语义

当一个 Worker 成功、另一个 Worker 失败时，Synthesis Layer 的行为：

| 场景 | 处理方式 |
|------|---------|
| 单源查询，目标 Worker 失败 | 降级到关键词搜索；仍失败则返回"该数据源暂时不可用，请稍后重试" |
| 跨源查询，1/2 Worker 失败 | 用部分结果生成回答，标注"以下信息缺少 [代码/文档/SQL] 维度的结果" |
| 跨源查询，2/3 Worker 失败 | 保留成功 Worker 的结果，降级到 L1/L2；提示用户缩小查询范围 |
| 全部 Worker 失败 | 触发 L3 缓存兜底（v2）；v1 返回友好错误+建议联系管理员 |

### 跨源多跳查询 Trace 示例

用户问题：**"用户登录模块的 PRD 需求改了哪些内容？影响了哪些代码文件和数据库表？"**

```
Step 1: Supervisor — Query Understanding
  ├─ 意图分类: cross-source (文档 + 代码 + SQL)
  ├─ 实体抽取: {module: "用户登录", actions: ["PRD变更", "代码影响", "数据库影响"]}
  ├─ 输出: 独立 Worker 调用（Code 和 SQL 共享 Doc Worker 输出作上下文，但各自独立检索）
  └─ [延迟预算: 200ms]

Step 2: Doc Worker — 检索相关 PRD 文档
  ├─ BM25 查询: "用户登录" + "PRD" + "需求变更"
  ├─ 向量查询: BGE-M3 语义检索 "登录模块需求规格"
  ├─ RRF 融合 → Top 5 文档片段
  ├─ LLM 提取: 涉及的改动点列表、需求 ID (如 REQ-2024-0187)
  └─ 返回: {req_ids: ["REQ-2024-0187"], changes: [{desc: "增加OAuth2.0登录", ...}, ...]}
        [延迟预算: 1500ms]

Step 3: Code Worker & SQL Worker — 并行执行
  ├─ Code Worker:
  │   ├─ 输入: {req_ids: ["REQ-2024-0187"], module: "用户登录"}
  │   ├─ AST 感知搜索: grep commit log + 函数签名匹配 "REQ-2024-0187" OR "oauth"
  │   ├─ 代码嵌入检索: 语义搜索 "authentication login oauth" 相关函数
  │   ├─ 元数据过滤: 文件路径包含 "auth/" OR "login/"
  │   └─ 返回: [{file: "src/auth/oauth.py", funcs: ["login_oauth", "token_refresh"]}, ...]
  │        [延迟预算: 1200ms]
  │
  └─ SQL Worker (与 Code Worker 并行，不依赖 Code 输出):
      ├─ 输入: {req_ids: ["REQ-2024-0187"], module: "用户登录"}
      ├─ Schema RAG: 检索 "user auth token oauth" 相关表名和列名
      ├─ 生成验证 SQL: SELECT table_name FROM information_schema... （验证表存在）
      └─ 返回: [{table: "users", cols: ["id","email","oauth_provider","oauth_token"]},
                {table: "user_sessions", cols: [...]}, {table: "oauth_states", cols: [...]}]
           [延迟预算: 1500ms]

  → Code & SQL 并行执行，总耗时: max(1200, 1500) = 1500ms

Step 4: Synthesis — 汇总生成回答
  ├─ 输入: Doc结果 + Code结果 + SQL结果
  ├─ RRF 去重融合（跨源结果归一化排序）
  ├─ LLM 生成结构化回答（含引用标注: [PRD §3.2], [src/auth/oauth.py:42], [users 表]）
  └─ 返回给用户
        [延迟预算: 1500ms]

总延迟预算: 200(路由) + 1500(Doc) + 1500(Code∥SQL) + 1500(Synthesis) = 4700ms
加上网络 RTT (3-4 hop × ~10ms) ≈ 4740ms
目标: P95 < 5000ms ✓  |  P99 < 8000ms（允许 LLM 尾延迟）
```
